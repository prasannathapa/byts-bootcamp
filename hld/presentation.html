altitude

HLD vs LLD.

  • High-level design - the view from ten thousand feet: which services and datastores exist, and how they talk. Boxes and arrows.
  • Low-level design - what is inside one box: classes, methods, schema. That was days one and two.
  • Same discipline - boundaries and change - one altitude higher.
foundationshld vs lld
the thesis

Every network hop is a port.

  • In real Instagram, every boundary is a call across the network, between machines.
  • In the one you build, every one of those boundaries is a Java interface - a port.
  • One machine to a thousand is a change at one wiring point, not a rewrite. The design is identical; only what sits behind the port changes.
foundationsthe thesis
how to design a system

A method you can repeat.

Every case study today - and every design interview - walks this line. Skip the numbers and you are guessing; skip the bottleneck and you drew a diagram, not a design.

high-level designthe method
the atom

Client asks, server answers.

  • A client sends a request; a server holds the data and logic and returns exactly one response.
  • The set of requests it will answer is its API - the contract.
  • It is synchronous: the client waits. Most of scaling is keeping that wait short.
foundationsrequest / response
the contract

An API is a promise.

  • A named set of operations, their inputs and outputs - stable even as the code behind changes.
  • REST over HTTP is the common shape: a verb on a resource (a URL).
  • Design the API first - it is the boundary everyone else builds against.
foundationsapi
a sense of scale

Numbers every engineer should feel.

  • Memory is roughly a million times faster than a round-trip across the world.
  • This is why we cache, co-locate, and go async.
  • You do not memorise these - you grow a nose for "that will be slow."
foundationslatency
scalability

Scale up, or scale out.

  • Vertical (up): a bigger machine. Simple - until you hit the biggest box money buys, and it is still one point of failure.
  • Horizontal (out): more machines. Near-limitless and resilient - but only if the servers are stateless.
  • The whole game: turn "one big thing" into "many small identical things."
scalevertical vs horizontal
the first move out

A load balancer over stateless servers.

  • A load balancer spreads requests across many identical instances.
  • Works only if servers are stateless - no per-user memory - so any instance can serve any request.
  • State moves down into the shared stores: the database and the cache.
scaleload balancer
staying up

Availability, reliability, fault tolerance.

  • Reliability - it does the right thing, correctly, over time.
  • Availability - it is up when asked, measured in nines.
  • Fault tolerance - a part fails and it keeps serving, via redundancy and no single point of failure.
99%→3.65 days/yr down
99.9%→8.8 hours/yr
99.99%→52 minutes/yr
99.999%→5 minutes/yr

each nine is ~10× more engineering

scalethe nines
one deployable

The monolith.

  • One codebase, one process, one deploy. Everything calls everything in memory.
  • Upside: simple to build, test, deploy, reason about. Fast calls, no network.
  • Downside: one bug can sink it all; you scale the whole thing even if only one part is hot.
foundationsmonolith
many deployables

Microservices.

  • Split by capability - each service owns its data, and deploys and scales alone.
  • Upside: independent scaling, independent teams, failure isolation.
  • The tax: the network is now in the middle - latency, partial failure, distributed transactions, ops. Not free.
foundationsmicroservices
the pragmatic middle

Start with a modular monolith.

  • One deployable, split inside into modules that talk only through ports.
  • The monolith's simplicity now, and the seams to split a service out later - when you actually feel the pain.
  • Exactly what you build tomorrow. Microservice-ready, without the tax up front.
foundationsmodular monolith
the front door

One gateway in front of many services.

  • A single entry point that routes each request to the right service.
  • The natural home for cross-cutting jobs: authentication, rate limiting, TLS, routing, aggregation.
  • One place to enforce policy - so every service does not reinvent it.
foundationsapi gateway
data

SQL or NoSQL?

SQL (relational)

Tables, joins, constraints, transactions. Strong integrity. Best when data is related and correctness matters - the core of most apps.

NoSQL

Key-value, document, wide-column, graph. Trades joins and strict schema for scale and flexibility. Best for huge, simple, or loosely-shaped data.

The honest answer is usually both, in different places. And because the store sits behind a repository port, moving one context to another database is an adapter swap, not a rewrite.

datasql vs nosql
store each fact once

Normalization: no duplication.

  • Each fact lives in exactly one place. A post references its author by author_id, not by copying the name.
  • Update once, and everyone sees it. No drift, no contradictions.
  • The cost: reading "post with its author" needs a join.
datanormalization
duplicate, on purpose

Denormalization: read fast.

  • Copy data so a read needs no join - store the author's handle right on the post, or a precomputed like_count.
  • The win: one cheap read instead of many joins - vital when reads outnumber writes 100:1.
  • The cost: now you must keep the copies in sync. You trade write-complexity for read-speed.
datadenormalization
find it fast

Indexes: the book's index.

  • Without one, the DB scans every row - O(n). An index (a B-tree) turns that into O(log n).
  • Index the columns you filter and sort on - here, (author_id, created_at) for the feed query.
  • The cost: every write must also update the index, and indexes take space. Index for your reads, not for everything.
dataindexing
the fundamental trade-off

CAP: under a partition, pick two.

  • When the network splits (a partition, and it will), a distributed store must choose:
  • Consistency - every read sees the latest write (refuse if unsure), or
  • Availability - always answer, even if the data is slightly stale.
  • Follows & likes tolerate staleness (AP). Login & payments do not (CP).
datacap theorem
copies for safety & reads

Replication: leader and followers.

  • Writes go to the leader; it streams changes to followers.
  • Reads spread across followers - great for read-heavy loads. If the leader dies, a follower is promoted.
  • The catch: replication lag - a follower can be a beat behind. That is eventual consistency in the wild.
datareplication
when one box is not enough

Sharding: split the data.

  • Partition rows across machines by a shard key - e.g. user_id % N, or a hash of it.
  • Each shard holds a slice, so no single box holds it all. This is how you scale writes.
  • The pain: cross-shard queries get hard, and a bad key makes a hot shard - the celebrity problem again.
datasharding
a fast store in front of a slow one

Cache: skip the slow path.

  • A small, fast store (memory / Redis) holding hot data. A hit returns in microseconds; a miss falls through to the database.
  • Read-heavy systems live or die by hit rate. Instagram's feed is mostly cache reads.
  • Common pattern: cache-aside - app checks cache, on miss loads the DB and fills the cache.
scalecache
the second hard problem

Invalidation & the stampede.

  • "There are only two hard things: naming, and cache invalidation." Stale data is the price of caching.
  • TTL (expire after N seconds) is simplest; write-through / explicit evict keeps it fresher.
  • Cache stampede: a hot key expires and thousands of requests hit the DB at once. Fix with jittered TTLs, a lock, or serving stale-while-revalidate.
datacache invalidation
big files, close to users

Blob storage and a CDN.

  • Photos and video do not belong in the database. They live in blob storage (S3); the DB keeps only the URL.
  • A CDN caches those bytes at edge locations near the user, so a photo loads from the same city, not across the world.
  • This is the single biggest lever for a media app's speed and bandwidth bill.
foundationscdn + blob
decouple the work

A message queue.

  • The writer drops an event on a queue and returns immediately. Workers consume it later, at their own pace.
  • Off the write path go the slow jobs: fan-out to feeds, notifications, hashtag indexing, thumbnails.
  • Buys you decoupling and a buffer for spikes - at the cost of eventual, not instant, results.
foundationsmessage queue
the fine print of async

At-least-once → be idempotent.

  • Queues usually promise at-least-once delivery - a message can arrive twice (retry after a crash).
  • So consumers must be idempotent: processing the same event twice has the same effect as once.
  • Do it with a dedupe key, an upsert, or a "already handled?" check. Exactly-once is mostly a myth you engineer around.
foundationsidempotency
protect the system

Rate limiting: cap the flood.

  • Cap how many requests a client makes per window - to stop abuse, runaway loops, and one noisy user starving everyone else.
  • Lives at the gateway: reject early with 429 Too Many Requests, before any work is done.
  • It is backpressure you choose, instead of a cascade you don't.
foundationsrate limiting
algorithms · windows

Fixed & sliding windows.

  • Fixed window - count per fixed slot (100/min), reset on the boundary. Dead simple; but a burst either side of the seam lets 2× through.
  • Sliding log - keep request timestamps, count the last 60s. Exact, but memory grows with traffic.
  • Sliding counter - weight the previous window by overlap. A cheap approximation that kills the boundary burst.
foundationswindow limiters
algorithms · buckets

Token bucket & leaky bucket.

  • Token bucket: refill r tokens/sec up to capacity B. Each request spends one; empty → reject. Allows a burst up to B, then steady r. What most APIs use.
  • Leaky bucket: requests queue and drain at a constant rate - output is perfectly smooth, bursts are buffered or dropped.
  • Token = bursty-but-bounded; leaky = smooth. Pick by whether downstream tolerates spikes.
foundationstoken / leaky bucket
across many gateways

Distributed rate limiting.

  • The problem: 10 gateway nodes, each with a local counter, = 10× the intended limit. Local state does not compose.
  • Centralize it in Redis: INCR key + EXPIRE is atomic and shared. Sliding window via a sorted set of timestamps; token bucket via a small Lua script.
  • The cost: a network hop per request, and Redis becomes a hot key / SPOF. Soften with a local pre-check + async sync, or shard the limit across keys.
foundationsdistributed limiters
you can't fix what you can't see

Logs, metrics, traces.

Logs

What happened - discrete events, with context. Search them when something broke.

Metrics

How much, how fast - counters and latencies over time. Alert on the golden signals: latency, traffic, errors, saturation.

Traces

One request's whole path across services - where the time actually went.

Every outage we're about to study was found - or missed - through these three.

back-of-the-envelope

Estimate before you architect.

  • Round aggressively - you want the order of magnitude, not a spreadsheet.
  • The numbers are the design: 15 TB/day of photos is why media never touches the DB.
  • Peak, not average, sizes your servers. Reads, not writes, shape your caches.
high-level designcapacity estimation
designing the contract

API design that scales.

Cursor pagination

Feeds page by an opaque cursor, not offset - stable when new posts arrive, and O(1) instead of O(offset).

Idempotency keys

A client-supplied key on POST /posts so a retry after a timeout doesn't double-post.

Versioning

/v1 in the path - evolve without breaking the apps already shipped.

Predictable shapes

Consistent resources, verbs, errors, and limits - and rate-limit headers so clients can back off.

war stories

How the big ones fell over.

The concepts so far are easier to remember as scars. Here are seven outages that took down large parts of the internet - each one is a slide you already have, learned the hard way.

The pattern is always the same: a small change, a shared dependency, and a blast radius nobody had drawn.

epic failuresintro
epic failure · 2017

AWS S3: one typo, half the internet.

  • What: an engineer running a routine playbook fat-fingered a command and removed far more S3 servers than intended in us-east-1.
  • Cascade: core subsystems had to full-restart - hours of downtime for thousands of sites that depended on S3.
  • Twist: AWS's own status dashboard was hosted on S3, so it couldn't even show red.
the lesson

Know your blast radius. Put guardrails on destructive tooling, cap how much one command can touch, and never host your status page on the thing that fails.

epic failuresaws s3 2017
epic failure · 2021

Meta: a config push erased them from the internet.

  • What: a routine BGP maintenance command withdrew the routes to Facebook's DNS servers. FB, Instagram, WhatsApp vanished for ~6 hours.
  • Root cause: the change looked local; its blast radius was global. With DNS unreachable, nothing resolved.
  • Twist: internal tools and badge access ran on the same network - engineers couldn't even get into the buildings to fix it.
epic failuresmeta bgp 2021
epic failure · 2012

Knight Capital: $440M in 45 minutes.

  • What: a deploy reused an old feature flag that switched on long-dead code - on one of eight servers.
  • Cascade: that server fired millions of erroneous stock orders. In 45 minutes it lost ~$440M and effectively ended the firm.
  • Root cause: a manual, inconsistent deploy and a flag repurposed without cleanup.
the lesson

Deploys must be automated and identical across nodes. Retire old flags. And have a kill switch - the difference between a bad minute and a bankruptcy is how fast you can stop.

epic failure · 2017

GitLab: rm -rf on production.

  • What: a tired engineer, fixing a replication issue at night, ran rm -rf on the primary database directory, thinking it was the replica.
  • The horror: they then found all five backup and replication methods were broken or empty.
  • Saved by: a chance 6-hour-old staging snapshot. They restored it - and livestreamed the whole recovery.
the lesson

A backup you have never restored does not exist. Test recovery regularly, and make destructive prod actions hard to do by accident.

epic failure · 2019

Cloudflare: one regex, 100% CPU, worldwide.

  • What: a new firewall rule contained a regular expression that backtracked catastrophically.
  • Cascade: it pinned every CPU core to 100% across the global fleet - worldwide 502s for ~27 minutes.
  • Root cause: the rule went everywhere at once, with no CPU budget to contain it.

epic failure · 2021

Fastly: one customer's config, a global CDN down.

  • What: a single customer pushed a valid config that tripped a latent bug shipped weeks earlier.
  • Cascade: ~85% of the CDN network errored - Reddit, Amazon, the NYT, gov.uk all went dark for about an hour.
  • Root cause: a dormant bug + shared infrastructure = one tenant can sink everyone.
the lesson

On shared infrastructure, you share fate. Validate configs, roll changes out gradually, and rehearse the fast rollback - Fastly's saving grace was recovering in ~an hour.

epic failure · 2021

Roblox: 73 hours, a slow cascade.

  • What: a three-day outage - not a single bang, but a slow collapse under load.
  • Root cause: a subtle contention bug in the service-discovery layer (Consul), triggered by a new feature's traffic, degraded the whole cluster.
  • Why so long: the system fought its own recovery - restarts made the contention worse.

the recurring outage

Twitter's fail whale.

  • What: 2007-2010, Twitter buckled constantly - the "fail whale" became a meme.
  • Root cause: a Rails monolith building each timeline on read (fan-out on read). As follows grew, that query got brutal.
  • The fix was not a patch - it was a re-architecture. Which is the perfect way into our next act.
the lesson

Your architecture has to match your scale. The right move here wasn't a bigger box - it was changing when the timeline is built. Systems don't just get built; they evolve.

epic failurestwitter
how we got here · 1

From bigger boxes to more boxes.

  • Early web: scale up - buy a bigger server. Simple, until you hit the ceiling, and it was still one point of failure.
  • The shift: scale out - many cheap commodity machines behind a load balancer.
  • The enabler was statelessness: push state into shared stores, and every box becomes disposable.
evolutionvertical → horizontal
how we got here · 2

Monolith → microservices → and back.

  • 2000s: big monoliths. 2010s: split into microservices for independent teams and scaling.
  • The overcorrection: five-engineer teams running thirty services, drowning in network calls and ops.
  • The backlash: in 2023 Amazon Prime Video moved a pipeline from microservices back to a monolith and cut cost ~90%. The modular monolith is fashionable again.
evolutionthe pendulum
how we got here · 3

The database pendulum.

  • RDBMS ruled - until web scale strained the single box.
  • NoSQL wave (~2007-12): Google's BigTable and Amazon's Dynamo papers spawned Cassandra, Mongo, HBase - drop joins and schema for scale.
  • NewSQL (Spanner, CockroachDB): SQL guarantees and horizontal scale.
  • Today: "just use Postgres" - it got so good most teams never needed to leave.
evolutiondb pendulum
how we got here · 4

Bare metal → VMs → containers → serverless.

  • Bare metal: one app per physical server, provisioned in weeks.
  • VMs: slice one machine into many - minutes, not weeks.
  • Containers (Docker '13) + Kubernetes: ship the app with its environment, packed densely.
  • Serverless / edge: don't manage servers at all - run a function, scale to zero.
evolutiondeployment
how we got here · 5

Twitter's timeline: pull → push → hybrid.

  • Pull (fan-out on read): build the timeline when you open the app. Simple - and it dies as follows grow (the fail whale).
  • Push (fan-out on write): when you post, write it into every follower's precomputed timeline. Reads become instant.
  • Hybrid: push for normal users; pull for the few celebrities with tens of millions of followers.
evolutiontwitter timeline
how we got here · 6

Instagram: huge scale, tiny stack.

  • Launched 2010 - 25,000 users on day one, millions within weeks.
  • Ran on boring, well-understood tech - Postgres, Django, Redis on AWS - with a famously tiny team serving tens of millions.
  • The lesson for tomorrow: fundamentals scale further than fashion.
the takeaway

You do not need thirty services to serve millions. You need clean seams (ports), a solid database, a cache, and a queue - which is exactly the modular monolith you build tomorrow.

design instagramthe method, applied
step 1 · requirements

What it must do.

Functional

Post a photo + caption · follow / unfollow · a feed of recent posts from people you follow · like & comment · search users & hashtags.

Non-functional

Massively read-heavy · photos are large, metadata is small · responsive from thousands to millions · the feed can be eventually consistent.

Pin these first - every later decision points back to "read-heavy" and "photos are large."

step 4 · data model

Five tables, a few deliberate copies.

Normalized core; like_count denormalized onto posts so a feed render needs no COUNT query.

design instagramdata model
step 5 · the hard part

How do you build the feed?

  • Pull (fan-out on read): at read time, query posts from everyone you follow and sort. Simple - and the join dies at scale. This was the fail whale.
  • Push (fan-out on write): when someone posts, write it into each follower's precomputed feed. Reads become a cheap lookup.
  • Given 100:1 reads, we pay at write time and make reads trivial. Push it is - mostly.
design instagramthe feed
step 5 · fan-out

Fan-out on write - except for celebrities.

  • On a post, push its id into each follower's feed list (in Redis). A feed read is then one lookup.
  • The celebrity problem: a post by someone with 50M followers = 50M writes. That spikes and lags.
  • Hybrid: push for normal accounts; for the few mega-accounts, followers pull their recent posts at read time and merge. Best of both.
design instagramcelebrity problem
step 6 · media

Photos: blob storage, served by CDN.

  • Upload the bytes to an object store (S3); the database keeps only the URL.
  • Serve every image from a CDN edge near the viewer - the biggest single speed lever.
  • Better still: the client uploads directly to blob via a presigned URL, so photo bytes never pass through your servers.
design instagrammedia + cdn
step 7 · async

The write path is event-driven.

  • POST /posts does the minimum synchronously - store the post, return the id.
  • Then it emits one PostCreated event. Off the write path, workers fan out to feeds, notify followers, index hashtags, make thumbnails.
  • The user waits for none of it. This is the message-queue seam - and tomorrow's Observer.
design instagramwrite path
step 8 · deep-dive

Cache the feed, shard the data.

Cache

Precomputed feeds live in Redis - a feed read is a cache hit. Hot post metadata too.

Shard

Partition posts and feeds by user_id. Index follows both ways for fan-out and follower lists.

Consistency

Feed & like-counts eventually consistent; follow & auth strong. CAP, chosen per feature.

Every port is a future service.

the same method, five more times

Five systems in five slides.

Every one of these is the same recipe - requirements, the one hard part, the key trick. The point: the toolkit generalises. Once you see the shape, you can design most things.

case studiesrapid-fire
case study · tinyurl

A URL shortener.

  • Hard part: generate a short, unique id - and make the redirect blazing fast.
  • Trick: a global counter → base62 encode it (7 chars = 3.5 trillion). No collisions, no lookups to check.
  • Read path: it is 99% reads - a key-value lookup (short → long), cached hard, then a 301 redirect.
case studiesurl shortener
case study · rate limiter

A rate limiter at scale.

  • You already designed this. Token bucket per client: refill r/s, capacity B, spend one per request.
  • At scale: the bucket state lives in Redis so every gateway node shares it - a small Lua script does check-and-decrement atomically.
  • Watch: the Redis round-trip and hot keys - soften with a local pre-check and per-key sharding.
callback

This is the afternoon's systems capstone block, if we build it - the algorithm you just saw, in code.

case studiesrate limiter
case study · whatsapp

A chat system.

  • Hard part: push, not pull - the server must reach the client instantly.
  • Trick: persistent WebSocket connections held open; a registry maps user → which server holds their socket.
  • Delivery: recipient online → push down their socket; offline → store and deliver on reconnect. Groups fan out.
case studieschat
case study · youtube / netflix

A video platform.

  • Hard part: one upload must play smoothly on any device and any connection.
  • Trick: an async transcode pipeline - one upload becomes many resolutions/bitrates (a queue + workers, exactly like our write path).
  • Delivery: chunks on a CDN; the player adapts bitrate to the network (HLS/DASH). The DB stores only metadata.
case studiesvideo
case study · search-as-you-type

Typeahead suggestions.

  • Hard part: return ranked suggestions in <100ms, on every keystroke.
  • Trick: a trie (prefix tree) held in memory; each node caches its top-k completions, ranked by popularity.
  • Scale: it is all reads from RAM; rebuild the ranked trie offline from logs, and cache hot prefixes at the edge.
