the one idea to carry all day

Every network hop is a port.

In real Instagram, each boundary between components is a call across the network. In tomorrow's build, each of those boundaries is a Java interface, a port.

So going from one machine to a thousand is a change at one wiring point, not a rewrite. That is what makes a modular monolith microservice-ready, and it is the thread through every slide today.

high-level designthe thesis
foundations

Client asks, server answers.

  • A client sends a request. A server holds the data and logic and sends back one response.
  • The contract the server exposes is its API. Each call is one request, one response.
  • Tomorrow the console is the client; the services are the server. Swap the console for REST controllers and it is a web app.
foundationsclient-server
the front door

One gateway in front of many services.

  • An API gateway is the single entry point. It routes each request to the right service.
  • It is also the natural home for the cross-cutting jobs: authentication, rate limiting, aggregation.
  • Tomorrow the gateway role is one class, the console front end. In production it sits in front of everything.
foundationsapi gateway
two extremes

Monolith or microservices?

Monolith: one deployable. Simple to build and run, but every change ships the whole thing, and one team can block another.

Microservices: many small, independently deployable services. Scale and ship each on its own, at the cost of network calls, and real operational complexity everywhere.

foundationsmonolith vs microservices
the middle path you will build

A modular monolith.

  • One process, but split inside into bounded contexts: user, post, feed, and so on.
  • They talk only through ports and an event bus, never by reaching into each other's tables.
  • Those seams are exactly where you would later cut real services. You get the simplicity of one box, and the option to split.
foundationsmodular monolith
the first move in scaling

Stateless servers behind a load balancer.

  • A load balancer spreads requests across many identical instances of the server.
  • This only works if the servers are stateless: they hold no per-user memory, so any instance can handle any request.
  • Tomorrow's services keep all state in the repositories, so N copies could run behind one balancer.
foundationsload balancer
data

SQL or NoSQL?

SQL (relational)

Tables, joins, constraints, transactions. Strong integrity. Great when data is related and correctness matters. Tomorrow's core: SQLite.

NoSQL

Key-value, document, wide-column, graph. Trades joins and strict schema for scale and flexibility. Great for huge, simple, or loosely-shaped data.

The honest answer is usually both, in different places. And because the store sits behind a repository port, moving one context to a different database is an adapter swap.

datasql vs nosql
store each fact once

Normalization: no duplication.

  • Each fact lives in exactly one place, linked by keys. Strong integrity, nothing to keep in sync.
  • A user's name lives in users. A post refers to its author by author_id, not by copying the name.
  • The cost: answering "show this post with its author" needs a join.
datanormalization
the deliberate opposite

Denormalization: duplicate, to read fast.

Counting a post's likes with a query every time is slow. So we keep a like_count right on the post, and bump it on each like.

The trade-off, and the whole lesson: the copy can drift. You accept brief staleness, and you pay to keep it in step on every write. Read speed, bought with write work.

datadenormalization
the trade-off you cannot escape

CAP: under a partition, pick two.

  • When the network splits, you can keep consistency or availability, not both.
  • Follows and login want to be correct: choose consistency.
  • The feed and the like-counts can be a little stale for a moment: choose availability. That is why they are denormalized and updated asynchronously.
datacap theorem
scaling the data itself

Replication and sharding.

  • Replication: keep copies on several nodes. One primary takes writes; replicas serve reads. Durability, plus read scaling.
  • Sharding: split one dataset across nodes by a key, like user_id % N, so no single node holds everything.
  • The hard part of sharding is cross-shard queries, like the social graph. Fan-out on write sidesteps it, which is our next topic.
datareplication + sharding
performance and scale

Cache: a fast store in front of a slow one.

  • Read-through: check the cache first. On a hit, return it. On a miss, read the database, put it in the cache, then return.
  • Hot data is served from memory; the database is spared the repeated work.
  • Tomorrow, feeds are cached read-through and invalidated when a new post arrives. The cache port is the Redis swap-in point.
scalecaching
where the photos actually live

Blob storage and a CDN.

  • Never put image bytes in your database. Store the file in blob storage, and keep only a media_url in the row.
  • A CDN is a fleet of edge servers that cache those files close to users, so the photo loads from a nearby city, not your one data centre.
  • Tomorrow, a post carries a media_url. In production that URL points at the CDN, not the app.
scaleblob + cdn
doing work later, and elsewhere

A message queue decouples the work.

  • A queue or event bus lets a producer announce that something happened and move on, without waiting.
  • Independent consumers react on their own. The producer does not know or care who is listening.
  • Tomorrow, one post publishes one event; feed, notifications, and search all react. Swap the in-process bus for Kafka and each becomes its own service.
scalemessage queue
the levers, together

Scale up, or scale out.

Vertical

A bigger machine. Simple, but capped, and a single point of failure.

Horizontal

More machines. Needs statelessness and sharding, but scales far. The whole ports design aims here.

Stay up

Availability, reliability, fault tolerance: bought with redundancy, replicas, retries, and queues.

Notice every lever, load balancer, replica, cache, queue, only works because the pieces talk through clean boundaries. Which is the design we build next.

now: design instagram

Start with what it must do.

Functional

Post a photo. Follow people. See a feed of posts from those you follow. Like, comment, search.

The shape of the load

Massively read-heavy: people scroll far more than they post. So we optimise reads, even at the cost of more work on writes.

Good design always starts here: what must it do, and what is the traffic actually like. Everything downstream follows from "read-heavy".

the data model

Normalized core, a few deliberate copies.

design instagramdata model
the one hard problem

How do you build a feed?

The naive way: at read time, find everyone you follow, gather all their recent posts, sort by time. A big join across follows and posts, run every single scroll.

For a read-heavy app with millions of users, that join is too slow, too often. We need reads to be cheap.

design instagramthe feed problem
the signature decision

Fan-out on write, not on read.

  • Do the work once, at write time. When you post, copy the post into every follower's own timeline, a materialized feed table.
  • Reading a feed is then a single indexed lookup. No join, no sort at read time. Reads are cheap, which is what a read-heavy app needs.
  • You moved the cost from the many reads to the one write. That is the trade.
design instagramfan-out on write
where the simple answer breaks

The celebrity problem.

  • A post by someone with a million followers is a million writes. That is write amplification.
  • The real fix is a hybrid: fan out on write for normal accounts, but for the few huge accounts, fan out on read.
  • At read time, merge your materialized feed with the handful of celebrities you follow, pulled live. Best of both.
design instagramcelebrity problem
one write, three reactions

The write path is event-driven.

  • PostService saves the post and publishes one PostCreated event. It knows nothing about feeds, notifications, or search.
  • Three independent consumers react: the feed fans out, notifications go to followers, search indexes the hashtags.
  • Adding a fourth reaction is a new subscriber, not an edit. That is open-closed, at the system level.

why it was worth it

Every port is a future service.

Because each boundary is an interface chosen at one wiring point, the local build and the distributed build are the same design.

Cache

in-memory today, Redis behind the same port tomorrow

Event bus

in-process today, Kafka behind the same port, and consumers become services

Repository

SQLite today, a sharded cluster behind the same port

That is the whole thesis paying off: you designed for scale without building for scale on day one.

What to carry out of today.

  • Stateless + load balancer: the basis of scaling out.
  • Normalize for truth, denormalize for speed, and pay the sync cost knowingly.
  • CAP: choose consistency or availability per feature.
  • Cache, CDN, queue: serve hot reads, media, and async work.
  • Fan-out on write: move cost to writes so reads stay cheap.
