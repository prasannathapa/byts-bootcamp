pillar 1 / encapsulation

What does one public field cost you?

It looks free. It is the most expensive line in the file.

encapsulationquestion
the answer

A public field is a promise you can never take back.

  • Anyone can read it and write it, from anywhere.
  • So you can never add a rule later, like "a balance can't go negative", without breaking every caller that touched it.
  • You use a microwave with a button. You do not open it and rewire the magnetron. That barrier is the feature.
encapsulationwhy
the move

Hide the state. Expose the behaviour.

exposed 

class Account {
    public double balance;
}
// anyone: acc.balance = -999;
guarded 

class Account {
    private double balance;
    void withdraw(double amt) {
        if (amt > balance) throw ...;
        balance -= amt;
    }
}

The private field plus the method is the whole idea: the rule lives in one place, and nobody can get around it.

encapsulationhow
the trap everyone falls into

Getters and setters are not encapsulation.

A setter for every field is just a public field with three extra lines. Ask what the object can do, not what fields it has.

Do not pull an object's data out to act on it. Tell the object to act. Tell, don't ask.

encapsulationtell, don't ask
the strongest version

State that cannot change cannot be corrupted.

If a value never changes after it is built, there is nothing to guard. Make fields final, offer no setters.

Java records give you an immutable value for free: fields, constructor, equals, all done.


record Money(long cents, String currency) { }
encapsulationimmutability
the smell, and the rule

How to spot a leak, and how to plug it.

Smells

Public fields. A setter for every field. Methods that hand out an internal list for the caller to mutate.

Rules

Private by default. Expose behaviour, not data. Copy or wrap anything you hand out.

pillar 2 / abstraction

A map is not the territory. So what do you leave off the map?

A map that shows every blade of grass is useless. Its value is what it leaves out.

abstractionquestion
the answer

Abstraction is deciding what to ignore.

  • A good abstraction shows the essentials and hides the mechanism.
  • You drive with a wheel and a pedal. You are not thinking about combustion.
  • The wheel is the abstraction: the simple idea you touch, standing in for the complicated thing underneath.
abstractionwhy
two ideas people confuse

Encapsulation hides data. Abstraction hides mechanism.

Encapsulation asks: who is allowed to touch this state? Abstraction asks: what is the simplest idea I can hand the caller, so they never see how it works?

Related, but different questions. You use both, constantly.

abstractionvs encapsulation
the move, in Java

Name what it can do. Say nothing about how.

An interface is a promise: these operations exist. It says nothing about the implementation.


interface PaymentGateway {
    Receipt charge(Money m);
}

// depend on the promise:
PaymentGateway gw =
    new StripeGateway();

Program to the interface. Swap the implementation, and the caller never notices.

Default to interfaces. Reach for an abstract class only when several types share real implementation, not just a shape.

abstractioninterface vs abstract
two hard-won warnings

Abstractions leak. And the wrong one is worse than none.

All non-trivial abstractions, to some degree, are leaky.

Joel Spolsky

Duplication is far cheaper than the wrong abstraction.

Sandi Metz

abstractionthe cost
the rule

Abstract carefully, and late.

  • Name the capability, not the class. PaymentGateway, not StripeHelper.
  • Return the interface, not the concrete type.
  • Do not abstract until you have seen the duplication at least twice. Guessing wrong is expensive.

pillar 3 / inheritance

The most dangerous tool in OOP. When is extends actually right?

It looks like free reuse. That is exactly why it bites.

inheritancequestion
why it bites

Inheritance welds a child to a parent's insides, forever.

  • A subclass depends not just on what the parent does, but on how.
  • Change the parent, and you can silently break every child. The fragile base class problem.
  • Classic trap: a Stack that extends ArrayList, so now anyone can add into the middle of your stack.
inheritancefragile base class
the test to apply

is-a, or has-a?

Use inheritance only for a true is-a that will never change: a Circle is a Shape.

Extending a class just to borrow its code is really a has-a. A Car is not an Engine; it has one.

inheritanceis-a vs has-a
the default you should reach for

Composition over inheritance.

inherits the guts 

class Stack extends ArrayList {
    // now callers can add(),
    // remove(), get(i) ... oops
}
holds what it needs 

class Stack {
    private List items;
    void push(x){ items.add(x); }
    Object pop(){ ... }
}

Hold the thing as a field and call it. You depend on its surface, not its insides, and you expose only what you mean to.

inheritancecomposition
when inheritance IS the right tool

A closed, known set of variants.

When a concept has a fixed family of shapes, inheritance is exactly right. Java's sealed types let you say so, and the compiler enforces it.


sealed interface Shape
    permits Circle, Square, Triangle { }

No rogue subclass can appear; a switch is checked for completeness.

inheritancesealed types
the rule

Compose by default. Inherit on purpose.

  • Default to composition.
  • Reach for inheritance only for a genuine, fixed is-a family.
  • Seal that family so it stays closed.
  • Never inherit just to reuse code.

pillar 4 / polymorphism

Different objects, same call. Why is that the whole point?

It is what lets you add new behaviour without editing old code.

polymorphismquestion
the answer

One call. Many behaviours. Chosen at runtime.

  • You call shape.area(). The right code runs based on what the object actually is.
  • The caller does not switch on type. The object already knows what to do.
  • That is dynamic dispatch, the engine under most OOP design.
polymorphismdynamic dispatch
the move, in Java

An interface, several implementations, one call.


interface Shape { double area(); }

record Circle(double r) implements Shape {
    public double area() { return Math.PI * r * r; }
}
record Square(double s) implements Shape {
    public double area() { return s * s; }
}

for (Shape s : shapes) total += s.area();   // no if, no switch

@Override is a contract you ask the compiler to check, not decoration.

polymorphismin java
why it matters

Open for extension. Closed for modification.

polymorphismopen/closed
a taste of tomorrow

Swap behaviour by swapping the object.

Hold an interface as a field, and you can change what a class does at runtime just by handing it a different implementation.

That is the Strategy pattern, which we build for real on Day 2. Polymorphism is the machinery underneath it.

polymorphismstrategy
the rule

If you are switching on a type, stop.

  • No switch or if chains on an object's type.
  • Program to the interface; let the object decide.
  • New behaviour is a new class, not a new branch in old code.
polymorphismchecklist

what to actually remember

The four pillars, one line each.

  • Encapsulation: hide state, expose behaviour, guard the rules in one place.
  • Abstraction: program to a simple idea; hide the mechanism behind it.
  • Inheritance: compose by default; inherit only a fixed is-a family, and seal it.
  • Polymorphism: let the object decide; add types, never edit callers.
