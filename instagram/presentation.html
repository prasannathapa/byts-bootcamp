session two
the decoupling leap

A new post must update the feed, notify followers, index hashtags. Who calls all three?

direct: the post service knows everyone 

void create(PostRequest r) {
    var post = repo.save(...);
    feed.fanOut(post);        // knows feed
    notifier.notify(post);    // knows notifier
    search.index(post);       // knows search
}
announce: it knows no one 

void create(PostRequest r) {
    var post = repo.save(...);
    events.publish(new PostCreated(post.id()));
}
// feed, notifier, search subscribe elsewhere

A fourth reaction on the left means editing the post service. On the right, it means adding a subscriber. The post service never changes again.

step-06 / events
one event, three reactions

This is the message-queue seam.

One PostCreated event. Three independent consumers react: the feed fans it out, the notifier alerts followers, search indexes the hashtags.

Today the bus is an in-process method call. Swap it for Kafka and those three consumers become three separate services. Nothing that publishes or subscribes changes.

step-06 / events
the honest cost

Events decouple. So what do they cost?

Traceability. With a direct call you read the stack and see what happens next. With an event, you have to know who subscribed.

  • Reach for events only when the publisher truly should not care who reacts.
  • Not everywhere. A plain method call is simpler and easier to follow.
  • We keep it honest by wiring every subscription in one visible place: the composition root.

If you cannot name what a pattern costs, you do not understand it yet.

design decision

swap the algorithm, keep the code

The same feed, two orders. How do you switch the algorithm at runtime?

chronological

Newest first

Sort by time. Simple, predictable, what people say they want.

engagement

Top posts first

Score by likes, comments, recency. What actually keeps people scrolling.

The feed service holds one FeedRankingStrategy. Hand it a different one and the whole feed reorders. How feeds are built and read never changes.

step-08 / strategy
the central scaling decision

When do you build a feed: when you post, or when you open the app?

We push on write. You open your feed far more often than you post, so pay once at write time and make every read instant.

step-08 / feed
where fan-out on write hurts

You have fifty million followers. You post once. What just happened?

Fifty million timeline writes, for one post. That is write amplification, and it does not scale.

Real systems go hybrid: fan-out on write for normal accounts, fan-out on read for the few huge ones, merged when you open the app. We build the write path and name the hybrid.

also honest

Following does not backfill.

Fan-out on write reaches only current followers. Follow someone new and you do not get their old posts. Real Instagram behavior, free from this design.

step-08 / feed
the trade at the heart of scale

Rendering a feed should never run COUNT(*). So where does the like count live?

normalized

One fact, one place

Likes live in a likes table. Correct, no duplication, easy to change.

denormalized

A copy, for speed

A like_count on the post and a materialized feed table, so reads never count or join.

The cost: every like updates two places, and the copy can briefly drift. That upkeep is the price of fast reads. Knowing when to pay it is most of systems design.

design decision

a real database enters

Five tables. What do we protect, and what do we deliberately duplicate?


users(id PK, username UNIQUE, follower_count)          -- follower_count: a copy
follows(follower_id, followee_id, PRIMARY KEY(both))   -- the graph, one edge per row
posts(id PK, author_id FK, type, like_count, comment_count)
likes(post_id, user_id, PRIMARY KEY(both))             -- the truth about likes
feed(user_id, post_id, created_at)                     -- the materialized timeline

Foreign keys keep it honest: you cannot like a post that does not exist. The counters and the feed table are the copies we accept for speed.

step-09 / sqlite
consistency versus availability

Must a like count be exact the instant you read it?

must be consistent

Follows, auth

Unfollow someone and it must take effect now. Correctness beats availability.

can be eventual

Feed, like counts

Off by one for a second is fine. Availability beats strict correctness.

That is why counters and feed can be denormalized and updated a beat later. CAP tells you which data you are allowed to be lazy about.

design decision
the seam pays off

We wrote the storage test once, at step 3. How many databases can it check?

Both. The SQLite adapter runs the exact same UserRepositoryContract.


class InMemoryUserRepositoryTest extends UserRepositoryContract { ... }
class SqliteUserRepositoryTest   extends UserRepositoryContract { ... }   // same tests, green

If SQLite behaved differently, the shared test would fail. Green on both proves they are interchangeable. Liskov substitution at the storage boundary, and the whole point of a port.

step-09 / sqlite
know your tool's limit

So why not just ship SQLite to production?

  • It is safe across processes, but single-writer: writes take a lock and serialize.
  • Two copies of the app on one file contend. That is why a single-writer database cannot scale writes across machines.
  • Worse in our code: the id generator is a per-process counter. Two instances mint the same ids and collide. Real systems use DB sequences, UUIDs, or Snowflake.

The lesson is not "SQLite is bad". It is that the port lets you swap it for Postgres the day you outgrow it.

design decision

step-10 / extras
the last seam

You open your feed twice in a row. Should we build it twice?

No. Read through a cache: a miss builds and stores, a hit returns instantly. A new post invalidates the affected feeds.

Today a hash map behind a Cache interface. Need Redis? Swap one adapter, the feed service does not change. You have seen this move four times now. That is the point.


List<Post> feed(long viewer) {
    return cache.get(viewer)
        .orElseGet(() -> {
            var built = build(viewer);
            cache.put(viewer, built);
            return built;
        });
}

the payoff
what to actually remember

The five ideas under everything.

  • Encapsulation is cognitive-load control, not security.
  • Compose by default; inherit only a closed, sealed set.
  • Depend on interfaces. Every seam is a future swap.
  • Announce events to decouple; accept the traceability cost.
  • Normalize the truth, denormalize the hot path, and know which data can lag ( CAP ).

Patterns are not the goal. They are named answers to forces. Learn the forces.
