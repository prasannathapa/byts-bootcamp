solidoverview
what SOLID actually is

Not five clever tricks. One goal.

Every principle serves a single aim: code you can change without fear.

Software that is hard to change is where projects go to die. SOLID is five specific, checkable rules that keep a change small and local, instead of a chain reaction across the whole system.

solidthe map
S / single responsibility

What makes a class need to change?

Not the number of methods it has. The number of people who can walk up and ask for one.

single responsibilityquestion
not what everyone says

Not "do one thing." Answer to one actor.

the misreading

"A class should do one thing."

Then ArrayList, with sixty methods, is a disaster. Yet nobody splits ArrayList. So that cannot be the rule.

the actual rule

Answer to one actor.

A "reason to change" is not a feature. It is the person or role whose request triggers the change. CFO, HR, DBA: those are reasons.

"A module should be responsible to one, and only one, actor."

ArrayList has one actor: the JDK collections team. Sixty methods, one source of change. Compliant.

single responsibilitythe actual rule
the actor lens

Find the people. Then find their code.

  • The canonical example: one Employee class, three methods, three departments.
  • calculatePay() answers to the CFO. reportHours() answers to HR. save() answers to the DBA.
  • Accounting tweaks the pay math, reuses a shared helper, and HR's report quietly breaks. Nobody warned HR.
single responsibilitythe actor lens
smell, and rule

Two teams in one file.

Smells

A class named Manager, Util, or Helper. A constructor with many collaborators. Imports from several unrelated namespaces. Two unrelated tickets touching it in one sprint.

Rule

One actor, one module. If two departments would edit this file for different reasons, it is two files in a trench coat.

Quick 30-second check: say what the class does in one sentence. If you need "and" (or a vague verb like "manages"), look again.

single responsibilitychecklist
O / open-closed

A new feature arrives. Do you edit old code, or add new?

Editing working code is how you break things that used to work.

open-closedquestion
the answer

Open for extension. Closed for modification.

open-closedthe answer
how, and the picture to keep

A stable interface; new behaviour behind it.

Think of a wall socket. The socket is fixed and standard: that is the closed part. Any appliance that fits the plug works: that is the open part.

In code, the socket is an interface. New behaviour is a new class that implements it. You extend the system by adding, and the old, tested code stays shut.

open-closedthe socket
smell, and rule

When a new case means editing old code.

Smell

A switch or if-else on a type that grows every time a variant is added. You keep reopening the same file.

Rule

Put the varying part behind an interface. A new case is a new class, not an edit to the old one.

open-closedchecklist
L / liskov substitution

A subtype should work anywhere its parent does. When does that break?

When the child looks like the parent, but does not behave like it.

liskov substitutionquestion
the classic trap

A Square is not a Rectangle.

  • Sounds true in maths. In code, a Rectangle promises: set width, and height is unchanged.
  • A Square that keeps its sides equal breaks that promise: set the width, and the height moves too.
  • Any code relying on the Rectangle's promise now breaks when handed a Square. That is an LSP violation.
liskov substitutionrectangle / square
the real rule

Inheriting the type means inheriting the promises.

  • A subtype must not demand more than the parent (no stronger preconditions).
  • A subtype must not deliver less than the parent (no weaker guarantees, no surprise exceptions).
  • If a subclass has to break the parent's behaviour to exist, it is not really that type. Reach for composition instead.
liskov substitutionthe contract
smell, and rule

The subclass that fights its parent.

Smells

An override that throws "not supported". Callers that check instanceof to special-case a subtype. A subclass that quietly does nothing.

Rule

A subtype must be usable through the parent's type with no surprises. If it cannot, do not subclass.

liskov substitutionchecklist
I / interface segregation

One big interface, or many small ones?

A big interface forces classes to depend on methods they never use.

interface segregationquestion
the answer

Many small, role-based interfaces.

A fat Machine interface with print, scan, and fax forces a simple printer to implement scan and fax, throwing on both.

Split it by role. A printer implements only Printer. Clients depend on the slice they use, nothing more.

interface segregationthe answer
smell, and rule

Methods you have to fake.

Smell

Implementers that throw or no-op half an interface. A client dragged into recompiling because a method it never calls changed.

Rule

Interfaces belong to their clients. Keep them small and role-shaped. A class can implement several.

interface segregationchecklist
D / dependency inversion

Should your business logic depend on the database?

If it does, you cannot change the database, or test the logic, without pain.

dependency inversionquestion
the answer

Both sides depend on an abstraction.

  • High-level policy should not reach down to low-level detail.
  • Instead, the policy defines an interface it needs; the detail implements it.
  • Now the arrow from the detail points up at the abstraction. That is the inversion.
dependency inversionthe answer
how you actually do it

Inject the dependency. Do not reach for it.

The policy never says new StripeGateway(). It receives a PaymentGateway in its constructor and uses it.

Now you can swap Stripe for PayPal, or a fake in a test, without touching the policy. This is the exact seam we lean on in the Day-4 project, everywhere.

dependency inversioninjection
smell, and rule

When business logic imports a database driver.

Smell

A new of a concrete service inside business code. Core logic importing a specific database or HTTP library.

Rule

Depend on an interface the policy owns. Pass the concrete thing in from the outside.

dependency inversionchecklist
how the five work together

One goal, five moves.

SRP

splits the system into small, single-purpose pieces

OCP + DIP

put interfaces at the seams, so pieces plug in without edits

LSP

keeps those plug-in pieces honest and substitutable

ISP

keeps the seams small, so nothing depends on more than it needs

And notice: OCP and DIP are just this morning's polymorphism and abstraction, applied with discipline. LSP is the inheritance contract. SOLID is OOP, done on purpose.

Capstone

From a switch to a seam.

  • Define a small ReportFormat interface: one method, format. (ISP)
  • A CsvFormat and a JsonFormat implement it. (OCP, LSP)
  • ReportService takes a ReportFormat and uses it; it never knows the concrete type. (DIP)
  • Formatting, validation, and data each live apart. (SRP)
capstonethe shape
Capstone

Which principle did each test just prove?

  • Passing your own new format to the service, and it just works: DIP and OCP.
  • Adding a third format without touching the service: open for extension.
  • Every format honouring the same one-method contract: ISP and LSP.
  • The service validating, not formatting; the formatter formatting, not saving: SRP.

what to actually remember

SOLID, one line each.

  • Single responsibility: one actor, one reason to change.
  • Open-closed: extend by adding, not by editing.
  • Liskov: a subtype must keep the parent's promises.
  • Interface segregation: small, role-shaped interfaces.
  • Dependency inversion: depend on abstractions, inject the details.
