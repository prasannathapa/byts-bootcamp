bonus patternsthe map
behavioral · command

Wrap an action as an object.

  • The pain: you want to trigger, queue, log - and above all undo and redo - actions, without the caller knowing their details.
  • The shape: an interface with run() and undo(). Each command captures what it needs to do its work and to reverse it. Keep an undo stack and a redo stack.

In the editor · CP1

InsertText is a Command. Editor.run executes and records it; undo pops and reverses; redo replays. Real undo and redo.

bonus patternscommand
behavioral · memento

Snapshot state, restore it later.

  • The pain: undo needs the previous state back - but you don't want to leak the object's internals to get it.
  • The shape: the object hands out an opaque snapshot, and later restores itself from one. Only it knows what is inside; everyone else just holds the token.

In the editor · CP2

Editor.snapshot() captures the whole buffer; restore(snapshot) puts it back. Pairs naturally with Command to power undo.

bonus patternsmemento
behavioral · state

Behaviour that changes with mode.

  • The pain: what an action does depends on a mode, and the code is drowning in if (mode == ...) branches everywhere.
  • The shape: make each mode an object behind one interface. The context delegates to its current state; a state can hand the context its successor.

In the editor · CP3

Mode is INSERT vs READONLY. The same keystroke inserts in one mode and is ignored in the other - the mode decides, not an if-ladder.

bonus patternsstate
structural · composite

Treat a tree like a single thing.

  • The pain: you have a part-whole hierarchy - a document is sections of paragraphs - and want to treat a leaf and a whole branch the same way.
  • The shape: one Node interface. Leaves and containers both implement it; a container holds child Nodes, and an operation just recurses.

In the editor · CP4

Paragraph is a leaf; Section holds Nodes. A whole document is just a Node - so any code that takes a Node takes the entire tree.

bonus patternscomposite
behavioral · iterator

Walk it without knowing its shape.

  • The pain: you want to visit everything in a structure, in order, without the caller learning that it is really a tree.
  • The shape: expose a sequence - Java's Iterable / Iterator, or hasNext() / next(). The caller loops; the traversal stays hidden.

In the editor · CP5

Walk every Paragraph depth-first, in document order. To the reader the tree looks like a flat list - exactly what a for-each wants.

bonus patternsiterator
behavioral · visitor

New operations, without touching the classes.

  • The pain: you keep needing new operations over a node tree - word count, spell-check, export - and editing every Node class each time is miserable.
  • The shape: nodes accept(visitor) and call visitor.visitX(this) - double dispatch. A new operation is a new visitor; the nodes never change.

In the editor · CP6

A WordCountVisitor tallies words across the tree. Worth knowing: Visitor makes new operations easy and new node types hard.

bonus patternsvisitor
behavioral · chain of responsibility

A pipeline of handlers.

  • The pain: a request should pass through a series of independent steps, each doing its bit or handing it on - and you want to add, drop, or reorder steps freely.
  • The shape: each handler holds the next one; it does its work, then delegates down the chain. Order is just wiring.

In the editor · CP7

A text-filter pipeline: trim, then collapse spaces, then censor. Each TextFilter transforms the text and passes it on.

bonus patternschain of responsibility
the seven we only name

Rarer in backend work - recognise them, don't force them.

Abstract Factory

Make whole families of related objects together (a matching theme).

Prototype

Clone a configured object instead of building a new one from scratch.

Bridge

Split an abstraction from its implementation so both can vary.

Flyweight

Share one immutable copy of repeated data (glyphs) to save memory.

Proxy

A stand-in that controls access - lazy, remote, or a guard.

Mediator

Objects talk through a central hub, not directly to each other.

Interpreter - represent a little language as a tree and evaluate it - rounds out the twenty-three. Look any of these up when you meet the problem; never go hunting for a place to use one.
