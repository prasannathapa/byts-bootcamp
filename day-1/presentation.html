day 1overview
first, the point of all of it

Two names for one goal.

OOP exists to manage complexity: cut a big program into objects with boundaries. SOLID exists to keep those boundaries sharp, so you can change code without fear.

day 1the point
pillar 1 / encapsulation

What does one public field cost you?

It looks free. It is the most expensive line in the file.

encapsulationquestion
the answer

A public field is a promise you can never take back.

  • Anyone can read it and write it, from anywhere.
  • So you can never add a rule later, like "a balance can't go negative", without breaking every caller that touched it.
  • You use a microwave with a button. You do not open it and rewire the magnetron. That barrier is the feature.
encapsulationwhy
the move

Hide the state. Expose the behaviour.

exposed 

class Account {
    public double balance;
}
// anyone: acc.balance = -999;
guarded 

class Account {
    private double balance;
    void withdraw(double amt) {
        if (amt > balance) throw ...;
        balance -= amt;
    }
}

The private field plus the method is the whole idea: the rule lives in one place, and nobody can get around it.

encapsulationhow
the trap everyone falls into

Getters and setters are not encapsulation.

A setter for every field is just a public field with three extra lines. Ask what the object can do, not what fields it has.

Do not pull an object's data out to act on it. Tell the object to act. Tell, don't ask.

encapsulationtell, don't ask
the strongest version

State that cannot change cannot be corrupted.

If a value never changes after it is built, there is nothing to guard. Make fields final, offer no setters.

Java records give you an immutable value for free: fields, constructor, equals, all done.


record Money(long cents, String currency) { }
encapsulationimmutability
the smell, and the rule

How to spot a leak, and how to plug it.

Smells

Public fields. A setter for every field. Methods that hand out an internal list for the caller to mutate.

Rules

Private by default. Expose behaviour, not data. Copy or wrap anything you hand out.

pillar 2 / abstraction

A map is not the territory. So what do you leave off the map?

A map that shows every blade of grass is useless. Its value is what it leaves out.

abstractionquestion
the answer

Abstraction is deciding what to ignore.

  • A good abstraction shows the essentials and hides the mechanism.
  • You drive with a wheel and a pedal. You are not thinking about combustion.
  • The wheel is the abstraction: the simple idea you touch, standing in for the complicated thing underneath.
abstractionwhy
two ideas people confuse

Encapsulation hides data. Abstraction hides mechanism.

Encapsulation asks: who is allowed to touch this state? Abstraction asks: what is the simplest idea I can hand the caller, so they never see how it works?

Related, but different questions. You use both, constantly.

abstractionvs encapsulation
the move, in Java

Name what it can do. Say nothing about how.

An interface is a promise: these operations exist. It says nothing about the implementation.


interface PaymentGateway {
    Receipt charge(Money m);
}

// depend on the promise:
PaymentGateway gw =
    new StripeGateway();

Program to the interface. Swap the implementation, and the caller never notices.

abstractioninterfaces
which tool, when

Interface or abstract class?

InterfaceAbstract class
Isa capabilitya partial family member
Statenoneshared fields ok
How manyimplement severalextend only one

Default to interfaces. Reach for an abstract class only when several types share real implementation, not just a shape.

abstractioninterface vs abstract
two hard-won warnings

Abstractions leak. And the wrong one is worse than none.

All non-trivial abstractions, to some degree, are leaky.

Joel Spolsky

Duplication is far cheaper than the wrong abstraction.

Sandi Metz

abstractionthe cost
the rule

Abstract carefully, and late.

  • Name the capability, not the class. PaymentGateway, not StripeHelper.
  • Return the interface, not the concrete type.
  • Do not abstract until you have seen the duplication at least twice. Guessing wrong is expensive.
abstractionchecklist
pillar 3 / inheritance

The most dangerous tool in OOP. When is extends actually right?

It looks like free reuse. That is exactly why it bites.

inheritancequestion
why it bites

Inheritance welds a child to a parent's insides, forever.

  • A subclass depends not just on what the parent does, but on how.
  • Change the parent, and you can silently break every child. The fragile base class problem.
  • Classic trap: a Stack that extends ArrayList, so now anyone can add into the middle of your stack.
inheritancefragile base class
the test to apply

is-a, or has-a?

Use inheritance only for a true is-a that will never change: a Circle is a Shape.

Extending a class just to borrow its code is really a has-a. A Car is not an Engine; it has one.

inheritanceis-a vs has-a
the default you should reach for

Composition over inheritance.

inherits the guts 

class Stack extends ArrayList {
    // now callers can add(),
    // remove(), get(i) ... oops
}
holds what it needs 

class Stack {
    private List items;
    void push(x){ items.add(x); }
    Object pop(){ ... }
}

Hold the thing as a field and call it. You depend on its surface, not its insides, and you expose only what you mean to.

inheritancecomposition
when inheritance IS the right tool

A closed, known set of variants.

When a concept has a fixed family of shapes, inheritance is exactly right. Java's sealed types let you say so, and the compiler enforces it.


sealed interface Shape
    permits Circle, Square, Triangle { }

No rogue subclass can appear; a switch is checked for completeness.

inheritancesealed types
the rule

Compose by default. Inherit on purpose.

  • Default to composition.
  • Reach for inheritance only for a genuine, fixed is-a family.
  • Seal that family so it stays closed.
  • Never inherit just to reuse code.
inheritancechecklist
pillar 4 / polymorphism

Different objects, same call. Why is that the whole point?

It is what lets you add new behaviour without editing old code.

polymorphismquestion
the answer

One call. Many behaviours. Chosen at runtime.

  • You call shape.area(). The right code runs based on what the object actually is.
  • The caller does not switch on type. The object already knows what to do.
  • That is dynamic dispatch, the engine under most OOP design.
polymorphismdynamic dispatch
the move, in Java

An interface, several implementations, one call.


interface Shape { double area(); }

record Circle(double r) implements Shape {
    public double area() { return Math.PI * r * r; }
}
record Square(double s) implements Shape {
    public double area() { return s * s; }
}

for (Shape s : shapes) total += s.area();   // no if, no switch

@Override is a contract you ask the compiler to check, not decoration.

polymorphismin java
why it matters

Open for extension. Closed for modification.

Keep this slide in mind. In an hour it becomes the O in SOLID.

polymorphismopen/closed
a taste of tomorrow

Swap behaviour by swapping the object.

Hold an interface as a field, and you can change what a class does at runtime just by handing it a different implementation.

That is the Strategy pattern, which we build for real on Day 2. Polymorphism is the machinery underneath it.

polymorphismstrategy
the rule

If you are switching on a type, stop.

  • No switch or if chains on an object's type.
  • Program to the interface; let the object decide.
  • New behaviour is a new class, not a new branch in old code.
polymorphismchecklist

day 1pillars synthesis
from pillars to principles

You have the tools. Now, five habits to keep them sharp.

SOLID is five specific, checkable rules in service of one goal: code you can change without fear.

None of them are new. Two of them are literally the pillars you just met — polymorphism and abstraction — stated as design rules. SOLID is OOP done on purpose.

solidthe point
the five, at a glance

S O L I D.

Awful names. By the end, each is one plain sentence plus one smell you can spot on sight.

solidthe map
S / single responsibility

What makes a class need to change?

Not the number of methods it has. The number of people who can walk up and ask for one.

single responsibilityquestion
the outdated definition

"A class should do one thing" is wrong.

Run the test. Open ArrayList in the JDK: about sixty methods — add, remove, get, set, sort.

If the rule were "do one thing," ArrayList would be a disaster. Nobody splits ArrayList.

So "do one thing" cannot be the rule. A count of methods tells you nothing.

single responsibilitythe myth
the actual rule

Answer to one actor.

"A module should be responsible to one, and only one, actor."

Robert C. Martin, correcting fifteen years of misquotes

  • A "reason to change" is not a feature. It is a person, a role, a stakeholder whose request triggers the change.
  • CFO, HR, DBA — those are reasons. Bad classes are not bad for having many methods; they are bad when their methods answer to different people who never talk.

Hold that thought over the break. After it, we watch one class serve three masters — and blow up.

single responsibilityone actor
part 2 / the rest of SOLID

Find the people. Then find their code.

  • One Employee class, three methods, three departments.
  • calculatePay() answers to the CFO. reportHours() answers to HR. save() answers to the DBA.
  • Accounting tweaks the pay math, reuses a shared helper — and HR's report quietly breaks. Nobody warned HR.
single responsibilitythe actor lens
before / after

Split the class along the actors.

three actors, one class 

class Employee {
  // CFO's rules
  Money calculatePay() { ... regularHours() ... }
  // HR's report
  String reportHours() { ... regularHours() ... }
  // DBA's schema
  void save() { ... }
  // shared helper = the hidden landmine
  int regularHours() { ... }
}
one actor each 

record Employee(String id, int worked) { }

class PayCalculator {        // CFO
  Money pay(Employee e) { ... }
}
class HourReporter {         // HR
  String report(Employee e) { ... }
}
class EmployeeRepo {         // DBA
  void save(Employee e) { ... }
}

Now the CFO's change touches only PayCalculator. HR's report can't break from a payroll edit — the shared helper is gone.

single responsibilitybefore / after
smell, and rule

Two teams in one file.

Smells

A class named Manager, Util, or Helper. A constructor with many collaborators. Imports from several unrelated namespaces. Two unrelated tickets touching it in one sprint.

Rule

One actor, one module. If two departments would edit this file for different reasons, it is two files in a trench coat.

Quick 30-second check: say what the class does in one sentence. If you need "and" (or a vague verb like "manages"), look again.

single responsibilitychecklist
O / open-closed

A new feature arrives. Do you edit old code, or add new?

Editing working code is how you break things that used to work.

open-closedquestion
the answer

Open for extension. Closed for modification.

open-closedthe answer
before / after

From a switch to an interface.

closed to nothing 

String export(Report r, String type) {
  switch (type) {
    case "csv":  return toCsv(r);
    case "json": return toJson(r);
    // add "xml"? reopen this method,
    // and every other switch like it
  }
  throw new IllegalArgumentException(type);
}
closed for modification 

interface ReportFormat {
  String format(Report r);
}
class CsvFormat  implements ReportFormat { ... }
class JsonFormat implements ReportFormat { ... }

// a new format is a NEW class;
// export() is never reopened
String export(Report r, ReportFormat f) {
  return f.format(r);
}

This is exactly the capstone you build later. Notice it is just the pillar-4 open/closed slide, now stated as a rule.

open-closedbefore / after
the picture, and the rule

A wall socket: fixed shape, endless appliances.

The socket is standard and closed. Any appliance whose plug fits just works — and you can invent new appliances forever. In code, the socket is an interface.

Smell

A switch or if-else on a type that grows a branch every time a variant is added. You keep reopening the same file.

Rule

Put the varying part behind an interface. A new case is a new class, not an edit to the old one.

open-closedthe socket
L / liskov substitution

A subtype should work anywhere its parent does. When does that break?

When the child looks like the parent, but does not behave like it.

liskov substitutionquestion
the classic trap

A Square is not a Rectangle.

  • Sounds true in maths. In code, a Rectangle promises: set width, and height is unchanged.
  • A Square that keeps its sides equal breaks that promise: set the width, and the height moves too.
  • Any code relying on the Rectangle's promise now breaks when handed a Square. That is an LSP violation.
liskov substitutionrectangle / square
before / after

The subtype that fails the parent's test.

looks right, lies 

class Rectangle {
  int w, h;
  void setW(int w){ this.w = w; }
  void setH(int h){ this.h = h; }
  int area(){ return w * h; }
}
class Square extends Rectangle {
  void setW(int w){ this.w = this.h = w; } // !
  void setH(int h){ this.w = this.h = h; } // !
}
void check(Rectangle r){
  r.setW(5); r.setH(4);
  assert r.area() == 20;  // FAILS for Square
}
don't force the is-a 

sealed interface Shape
    permits Rectangle, Square {
  int area();
}
record Rectangle(int w, int h) implements Shape {
  public int area(){ return w * h; }
}
record Square(int side) implements Shape {
  public int area(){ return side * side; }
}
// no inheritance to abuse; each keeps
// its own honest promise

If a subclass can only exist by breaking the parent's behaviour, it is not really that type. This is the inheritance contract from Part 1, enforced.

liskov substitutionbefore / after
the real rule, and the smell

Inherit the type, inherit the promises.

  • A subtype must not demand more than the parent (no stronger preconditions).
  • A subtype must not deliver less than the parent (no weaker guarantees, no surprise exceptions).

Smells

An override that throws "not supported". Callers that check instanceof to special-case a subtype. A subclass that quietly does nothing.

Rule

A subtype must be usable through the parent's type with no surprises. If it cannot, do not subclass — compose instead.

liskov substitutionthe contract
I / interface segregation

One big interface, or many small ones?

A big interface forces classes to depend on methods they never use.

interface segregationquestion
the answer

Many small, role-based interfaces.

A fat Machine interface with print, scan, and fax forces a simple printer to implement scan and fax, throwing on both.

Split it by role. A printer implements only Printer. Clients depend on the slice they use, nothing more.

interface segregationthe answer
before / after

Stop faking methods you don't have.

forced to lie 

interface Machine {
  void print(Doc d);
  void scan(Doc d);
  void fax(Doc d);
}
class SimplePrinter implements Machine {
  public void print(Doc d){ ... }
  public void scan(Doc d){
    throw new UnsupportedOperationException();
  }
  public void fax(Doc d){
    throw new UnsupportedOperationException();
  }
}
role-shaped 

interface Printer { void print(Doc d); }
interface Scanner { void scan(Doc d); }
interface Fax     { void fax(Doc d); }

class SimplePrinter implements Printer {
  public void print(Doc d){ ... }
}
class OfficeMachine
    implements Printer, Scanner, Fax {
  ...   // implement several when you are several
}

Dead, lying methods vanish. And a change to Fax can no longer force SimplePrinter to recompile.

interface segregationbefore / after
smell, and rule

Methods you have to fake.

Smell

Implementers that throw or no-op half an interface. A client dragged into recompiling because a method it never calls changed.

Rule

Interfaces belong to their clients. Keep them small and role-shaped. A class can implement several.

interface segregationchecklist
D / dependency inversion

Should your business logic depend on the database?

If it does, you cannot change the database, or test the logic, without pain.

dependency inversionquestion
the answer

Both sides depend on an abstraction.

  • High-level policy should not reach down to low-level detail.
  • Instead, the policy defines an interface it needs; the detail implements it.
  • Now the arrow from the detail points up at the abstraction. That is the inversion.
dependency inversionthe answer
before / after

Inject the dependency. Don't reach for it.

welded to Stripe 

class OrderService {
  // policy reaches down to a concrete detail
  private final StripeGateway gw
      = new StripeGateway();

  void checkout(Order o){
    gw.charge(o.total());
  }
}
// can't swap Stripe; can't test without it
depends on a promise 

interface PaymentGateway { void charge(Money m); }

class OrderService {
  private final PaymentGateway gw;
  OrderService(PaymentGateway gw){   // inject
    this.gw = gw;
  }
  void checkout(Order o){ gw.charge(o.total()); }
}
class StripeGateway implements PaymentGateway { ... }
// tests pass a fake; swap Stripe -> PayPal freely

The interface passed in from the outside is the seam we lean on everywhere in the Day-4 project — and in this afternoon's capstone.

dependency inversionbefore / after
smell, and rule

When business logic imports a database driver.

Smell

A new of a concrete service inside business code. Core logic importing a specific database or HTTP library.

Rule

Depend on an interface the policy owns. Pass the concrete thing in from the outside.

dependency inversionchecklist
how the five work together

One goal, five moves.

SRP

splits the system into small, single-purpose pieces

OCP + DIP

put interfaces at the seams, so pieces plug in without edits

LSP

keeps those plug-in pieces honest and substitutable

ISP

keeps the seams small, so nothing depends on more than it needs

OCP and DIP are this morning's polymorphism and abstraction, applied with discipline. LSP is the inheritance contract. SOLID is OOP, done on purpose.

Part 2 · the report

From a switch to a seam.

  • Define a small ReportFormat interface: one method, format. (ISP)
  • A CsvFormat and a JsonFormat implement it. (OCP, LSP)
  • ReportService takes a ReportFormat and uses it; never knows the concrete type. (DIP)
  • Formatting, validation, and data each live apart. (SRP)
capstonepart 2 · the shape
Part 2 · debrief

Which principle did each test just prove?

  • Passing your own new format to the service, and it just works: DIP and OCP.
  • Adding a third format without touching the service: open for extension.
  • Every format honouring the same one-method contract: ISP and LSP.
  • The service validating, not formatting; the formatter formatting, not saving: SRP.
