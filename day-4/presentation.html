Day 3 solved · LRU cache1 / constraints

O(1) get. O(1) put. Evict the least-recently-used.

HashMap alone?

O(1) find - but a map has no order. "Who was touched longest ago?" is unanswerable without scanning everything.

List alone?

Perfect order - front = newest, tail = oldest. But finding a key means walking the list: O(n). Too slow.

Both, glued

Map: key → node (O(1) find). Doubly-linked list: recency order (O(1) unlink/relink). Each covers the other's blind spot.

The insight: the map does not store values - it stores pointers into the list. Once you hold the node, moving it is pure pointer surgery: no search, no shifting. That is where O(1) comes from.

public final class LruCache<K, V> {
    // ① the two structures, glued
    private final Map<K, Node<K,V>> map = new HashMap<>();
    private final Node<K,V> head = new Node<>(null, null); // sentinel: MRU side
    private final Node<K,V> tail = new Node<>(null, null); // sentinel: LRU side
    private final int capacity;

    private static final class Node<K, V> {
        K key; V val;
        Node<K,V> prev, next;                 // doubly linked
        Node(K k, V v) { key = k; val = v; }
    }

    public LruCache(int capacity) {
        this.capacity = capacity;
        head.next = tail;                     // empty list: HEAD <-> TAIL
        tail.prev = head;
    }


    // ③ unlink: neighbours point past n (2 writes)
    private void unlink(Node<K,V> n) {
        n.prev.next = n.next;
        n.next.prev = n.prev;
    }
    // ④ addFront: splice in right after HEAD (4 writes)
    private void addFront(Node<K,V> n) {
        n.next = head.next;
        n.prev = head;
        head.next.prev = n;
        head.next = n;
    }
    // ③+④ together = "touched → most recent"
    private void moveToFront(Node<K,V> n) {
        unlink(n);
        addFront(n);
    }

Why sentinels? HEAD and TAIL are dummy nodes that always exist - so unlink/addFront never null-check "am I first? am I last?". Every node always has both neighbours. Fewer branches, zero edge cases.

public V get(K key) {
    Node<K,V> n = map.get(key); // ② hash hop to the node
    if (n == null) return null; //    miss: not ours
    moveToFront(n);             // ③④ touched → front
    return n.val;
}


public void put(K key, V value) {
    Node<K,V> n = map.get(key);
    if (n != null) {                 // existing: update
        n.val = value;
        moveToFront(n);              // ③④ and refresh
        return;
    }
    n = new Node<>(key, value);
    map.put(key, n);                 // ⑤ enter BOTH
    addFront(n);                     //   structures
    if (map.size() > capacity) {
        Node<K,V> lru = tail.prev;   // ⑤ tail.prev IS the LRU
        unlink(lru);
        map.remove(lru.key);         //   forget in BOTH
    }
}

The discipline: every mutation touches both structures or neither - add to map + list together, evict from list + map together. Let them drift apart and you have a ghost entry. That symmetry is the whole implementation.

Day 3 solved · token bucket1 / constraints

Allow bursts, cap the average rate - with no background thread.

The guarantee

Bursts up to capacity tokens are fine; sustained traffic settles to r tokens/sec. Empty bucket → reject (HTTP 429).

The naive trap

A thread that adds a token every second? Now you own a timer, thread-safety, shutdown… for every single limiter. Thousands of limiters = thousands of timers.

The insight: lazy refill

Nobody adds tokens on a schedule. The next call computes elapsed-time × rate and credits them retroactively. Time itself does the bookkeeping.

And the clock is injected (a LongSupplier), not read from the system - so a test can say "two seconds passed" without sleeping two seconds. Deterministic tests, zero waiting.

public final class TokenBucket {
    private final double capacity;       // burst ceiling
    private final double refillPerMilli; // the rate r
    private final LongSupplier clock;    // INJECTED time
    private double tokens;               // fractional!
    private long last;                   // last refill moment

    public TokenBucket(double capacity,
                       double refillPerSecond,
                       LongSupplier clockMillis) {
        this.capacity = capacity;
        this.refillPerMilli = refillPerSecond / 1000.0;
        this.clock = clockMillis;
        this.tokens = capacity;          // ① start full
        this.last = clockMillis.getAsLong();
    }


    public boolean allow() {
        refill();                        // ④ settle up first
        if (tokens >= 1.0) {
            tokens -= 1.0;               // ② spend one
            return true;
        }
        return false;                    // ③ empty → 429
    }

    private void refill() {
        long now = clock.getAsLong();
        double add = (now - last) * refillPerMilli;
        if (add > 0) {                   // ④ elapsed × rate
            tokens = Math.min(capacity, tokens + add);
            last = now;                  // ⑤ capped at capacity
        }
    }
}

Recurring: the injected clock is dependency inversion on time - the same seam as every port today. The test hands in a fake clock, advances it by hand, and asserts exact behaviour. Watch for it again in Story this afternoon.

Day 3 solved · load balancer1 / constraints

Spread requests over N backends - and let the policy change.

The job

next() picks a backend for each request; release() is called when that request finishes. Backends can be added and removed live.

Two policies

Round-robin: fair by count, blind to load. Least-connections: route by live load - needs the active-connection book-keeping.

The design constraint

The policy must be swappable at runtime without touching the balancer. That is a textbook Strategy: one interface, N interchangeable algorithms.

Separation: the balancer owns the state (backends, active counts, call number); the strategy owns only the choice. It's a pure function: same inputs, same pick - which also makes it trivially testable.

@FunctionalInterface
public interface Strategy {   // the seam
    int choose(List<String> backends,
               int[] active, int callNumber);
}

public static final class RoundRobin implements Strategy {
    public int choose(List<String> b, int[] a, int call) {
        return call % b.size();          // ①② fair cycle
    }
}

public static final class LeastConnections implements Strategy {
    public int choose(List<String> b, int[] active, int call) {
        int best = 0;                    // ④ scan live load
        for (int i = 1; i < active.length; i++)
            if (active[i] < active[best]) best = i;
        return best;
    }
}


private final List<String> backends = new ArrayList<>();
private int[] active = new int[0];   // live connections
private Strategy strategy;           // swappable at runtime
private int calls = 0;

public void setStrategy(Strategy s) { this.strategy = s; }

public String next() {               // pick + book-keep
    int idx = strategy.choose(backends, active, calls++);
    active[idx]++;                   // one more in flight
    return backends.get(idx);
}

public void release(String backend) { // ⑤ request done
    int i = backends.indexOf(backend);
    if (i >= 0 && active[i] > 0) active[i]--;
}

Recurring (Day 2): Strategy, verbatim - and this afternoon the feed's ranking is the exact same shape. HLD boxes are LLD patterns wearing systems names.

Day 3 solved · LFU cache1 / constraints

Evict by how often, not how recently - still O(1).

LRU's blind spot

A key read 1000× gets evicted the moment it sits quiet while three one-hit wonders pass through. Frequency should count for something.

The rule

Evict the least-frequently-used; break ties by least-recently-used. And still O(1) - so no "scan for the smallest count".

The structure

Three maps: values, freq (key→count), and buckets (count→keys, a LinkedHashSet = recency order inside). Plus one integer: minFreq.

The insight: group keys into frequency buckets. Touching a key moves it one bucket up - O(1). Evicting reads buckets[minFreq]'s eldest - O(1). minFreq is the cursor that makes "find the smallest" free.

private final Map<K, V> values = new HashMap<>();
private final Map<K, Integer> freq = new HashMap<>();
private final Map<Integer, LinkedHashSet<K>> buckets = new HashMap<>();
private int minFreq = 0;

public V get(K key) {
    if (!values.containsKey(key)) return null;
    touch(key);                          // ② every hit bumps
    return values.get(key);
}

public void put(K key, V value) {
    if (values.containsKey(key)) {
        values.put(key, value);
        touch(key);                      // update counts too
        return;
    }
    if (values.size() >= capacity) evict();  // ④ make room FIRST
    values.put(key, value);
    freq.put(key, 1);                    // ⑤ newcomers start at 1
    buckets.computeIfAbsent(1, f -> new LinkedHashSet<>()).add(key);
    minFreq = 1;                         // ⑤ so minFreq resets
}


/** ② Bump a key one bucket up. */
private void touch(K key) {
    int f = freq.get(key);
    LinkedHashSet<K> b = buckets.get(f);
    b.remove(key);                       // leave old bucket
    if (b.isEmpty()) {
        buckets.remove(f);
        if (minFreq == f) minFreq = f + 1;  // ③ cursor follows
    }
    freq.put(key, f + 1);
    buckets.computeIfAbsent(f + 1,
            x -> new LinkedHashSet<>()).add(key);
}

/** ④ Drop the LFU key (LRU inside the min bucket). */
private void evict() {
    LinkedHashSet<K> b = buckets.get(minFreq);
    K victim = b.iterator().next();      // eldest = LRU tie-break
    b.remove(victim);
    if (b.isEmpty()) buckets.remove(minFreq);
    values.remove(victim);
    freq.remove(victim);                 // forget EVERYWHERE
}

Same discipline as LRU: three maps must move together, or you leak ghosts. And iterator().next() on a LinkedHashSet is "the eldest" - insertion order is the recency order, for free.

Day 3 solved · consistent hashing1 / constraints

Map keys to N cache nodes - without a reshuffle when N changes.

The naive answer: node = hash(key) % N. Works - until one node dies. Now N=3 became N=2, and almost every key maps somewhere new. The whole cache goes cold at once; the database takes the stampede.

The insight: stop tying keys to N. Place nodes on a circle of hash values; a key belongs to the first node clockwise from its own hash. Remove a node and only its arc of keys moves - about 1/N of them. The rest never notice.

And virtual nodes?

Three real nodes on a circle can land lopsided - one owns half the ring. So each node is placed at several hashed points ("A#0", "A#1", …). More points = smoother slices, and a removed node's load spreads over everyone, not one neighbour.

Where it lives

Memcached/Redis client sharding, Cassandra and DynamoDB partitioning, CDN routing - anywhere keys meet a changing fleet.

public final class HashRing {
    private final int replicas;         // virtual points per node
    private final SortedMap<Integer, String> ring
            = new TreeMap<>();          // point → node, SORTED

    public void addNode(String node) {  // ① N points per node
        for (int i = 0; i < replicas; i++)
            ring.put(hash(node + "#" + i), node);
    }

    public void removeNode(String node) { // ④ drop its points
        for (int i = 0; i < replicas; i++)
            ring.remove(hash(node + "#" + i));
    }


    /** ② first node clockwise from the key's hash */
    public String getNode(String key) {
        if (ring.isEmpty()) return null;
        int h = hash(key);
        SortedMap<Integer, String> tail = ring.tailMap(h);
        Integer point = tail.isEmpty()
                ? ring.firstKey()       // ③ wrap around
                : tail.firstKey();      // ② next point ahead
        return ring.get(point);
    }

    /** deterministic FNV-1a, kept non-negative */
    static int hash(String s) {
        int h = 0x811c9dc5;
        for (int i = 0; i < s.length(); i++) {
            h ^= s.charAt(i);
            h *= 0x01000193;
        }
        return h & 0x7fffffff;
    }
}

The load-bearing choice is the TreeMap: a sorted map gives tailMap(h) - "everything clockwise of me" - in O(log n). The "ring" is just a sorted map that wraps. Note we hash deterministically (FNV), never hashCode(), which can differ across JVMs.

Day 3 solved · Raft election (boss)1 / constraints

N nodes must agree on one leader - never two.

Why it's hard

Messages race, nodes crash, two nodes can stand for election at once. "Just pick the first" splits the brain: two leaders both accepting writes.

Raft's three rules

Time is split into numbered terms. A node grants at most one vote per term. A candidate needs a strict majority. A higher term always supersedes.

Why majority works

Two majorities of the same cluster must overlap in at least one node - and that node only had one vote to give. Two leaders in one term is mathematically impossible.

Scope honesty: ours is the safety core only - no timeouts, heartbeats, threads, or logs. Full Raft is a multi-week project; the voting rules below are its heart, in 70 lines.

public static final class Node {
    int term = 0;
    Integer votedFor = null;  // whom, THIS term
    Role role = Role.FOLLOWER;

    boolean requestVote(int candidateTerm, int candidateId) {
        if (candidateTerm < term) return false; // stale? refuse
        if (candidateTerm > term) {   // ⑤ newer term:
            term = candidateTerm;     //   catch up,
            votedFor = null;          //   vote resets,
            role = Role.FOLLOWER;     //   step down
        }
        if (votedFor == null || votedFor == candidateId) {
            votedFor = candidateId;   // ③ ONE vote per term
            return true;
        }
        return false;                 // already spent it
    }
}


/** One election round with {@code candidateId} standing. */
public int elect(int candidateId) {
    Node c = nodes.get(candidateId);
    c.term += 1;                      // ② new term
    c.role = Role.CANDIDATE;
    c.votedFor = c.id;                // ② votes for itself
    int votes = 1;
    for (Node other : nodes) {
        if (other.id == candidateId) continue;
        if (other.requestVote(c.term, c.id)) votes++;
    }
    if (votes > nodes.size() / 2) {   // ④ STRICT majority
        c.role = Role.LEADER;
        return c.id;
    }
    c.role = Role.FOLLOWER;           // failed: stand down
    return -1;
}

Read the guarantee out of the code: votedFor is nulled only when the term rises - so within one term a node physically cannot vote twice. Majority + one-vote-per-term = at most one leader per term. The proof fits in two if-statements.

    private final long id, authorId;          // references by ID, never object graphs
    private final String caption;
    private final LocalDateTime createdAt;
    private int likeCount, commentCount;      // PRIVATE: nobody else may touch

    public abstract PostType getType();

public interface EventBus {          // the PORT
    <T extends DomainEvent>
    void subscribe(Class<T> type, Consumer<T> handler);

    void publish(DomainEvent event);
}

public record PostCreated(
    long postId, long authorId,
    LocalDateTime createdAt
) implements DomainEvent {}          // facts, frozen


public class InProcessEventBus implements EventBus {
    private final Map<Class<?>, List<Consumer<?>>> handlers
            = new ConcurrentHashMap<>();

    public <T extends DomainEvent> void subscribe(
            Class<T> type, Consumer<T> handler) {
        handlers.computeIfAbsent(type,
            k -> new CopyOnWriteArrayList<>()).add(handler);
    }

    public void publish(DomainEvent event) {
        List<Consumer<?>> subs = handlers.get(event.getClass());
        if (subs == null) return;    // nobody cares - fine!
        for (Consumer<?> s : subs)   // call each, in order,
            ((Consumer<DomainEvent>) s).accept(event);  // same thread
    }
}

Recurring (Day 2): Observer, literally - a list of listeners, a loop that notifies. The only upgrade over the cafe's version: subscribers are grouped by event type.

step 08feed · the decision

The hard one: how do you build a feed?

Fan-out on read: when bob opens the app, query everyone he follows, merge, sort. Cheap writes - but the read grows with follow-count. That query was Twitter's fail-whale.

Fan-out on write: when alice posts, push the post id into every follower's precomputed timeline. Expensive write - but the read becomes a lookup.

The decision: we're read-heavy (non-functional req #1) - reads outnumber writes ~100:1. Pay at write time. And we already own the trigger: the PostCreated event from step 6.

The honest limit: a celebrity with 50M followers = 50M writes per post. Real Instagram goes hybrid - push for normal users, pull for celebrities. Ours pushes; we name the limit.

public class EngagementFeedStrategy
        implements FeedRankingStrategy {
    public List<Post> rank(List<Post> posts, LocalDateTime now) {
        return posts.stream()
            .sorted(comparingDouble(
                (Post p) -> score(p, now)).reversed())
            .toList();
    }
    private double score(Post p, LocalDateTime now) {
        double engagement = 3.0 * p.getLikeCount()
                          + 5.0 * p.getCommentCount();
        return engagement + recencyBoost(p, now);
    }
}

// PostService.like - idempotent by design
public boolean like(long postId, long userId) {
    Post post = require(postId);
    if (posts.addLike(postId, userId, now())) {
        // ① truth row inserted (PK blocked dupes)
        post.incrementLikes();   // ② bump the copy
        posts.save(post);        // ③ persist the copy
        return true;
    }
    return false;                // already liked: no-op
}

Why keep both? The likes table answers "who liked this?" and blocks double-likes via its PK. The counter answers "how many?" in zero joins. The service is the sync point - that upkeep is the price of denormalization, paid on the write because reads outnumber writes.

public <T extends Post> T save(T post) {
    String sql = "INSERT INTO posts(id, author_id, type, caption, media_url, filter, "
            + "duration_seconds, created_at, like_count, comment_count) VALUES(?,?,?,?,?,?,?,?,?,?) "
            + "ON CONFLICT(id) DO UPDATE SET caption=excluded.caption, "        // UPSERT: insert or update
            + "like_count=excluded.like_count, comment_count=excluded.comment_count";
    String mediaUrl = null; String filter = null; Integer duration = null;
    switch (post) {                                        // the sealed switch maps subtype -> columns
        case PhotoPost photo -> { mediaUrl = photo.getImageUrl(); filter = photo.getFilter(); }
        case VideoPost video -> { mediaUrl = video.getVideoUrl(); duration = video.getDurationSeconds(); }
        case TextPost ignored -> { }
    }
    try (PreparedStatement ps = connection.prepareStatement(sql)) {   // prepared = safe from injection
        ps.setLong(1, post.getId());  ps.setLong(2, post.getAuthorId());
        ps.setString(3, post.getType().name());  ps.setString(4, post.getCaption());
        ps.setString(5, mediaUrl);  ps.setString(6, filter);
        if (duration == null) ps.setNull(7, Types.INTEGER); else ps.setInt(7, duration);
        ps.setString(8, post.getCreatedAt().toString());
        ps.setInt(9, post.getLikeCount());  ps.setInt(10, post.getCommentCount());
        ps.executeUpdate();
    } catch (SQLException e) { throw new DataAccessException("save post", e); }
    return post;
}

What JDBC is: Java's standard driver interface - DriverManager.getConnection (Day 2's factory!), PreparedStatement, ResultSet. And note the sealed switch again: one Post hierarchy, three column mappings, checked by the compiler.

The feed read is the hottest path. Make it instant.

The decision: a read-through cache in front of the feed build, behind a Cache port. Hit → return cached. Miss → build, store, return. And because a cache is a second copy of the truth, every new post must invalidate the affected feeds.

Why cache the feed specifically? Read constantly, changes only when someone you follow posts, expensive to build (lookup + hydrate + rank). High reads × low change × costly build = the textbook cache candidate. (Yesterday's lecture; this morning's LRU.)

public List<Post> feed(long viewerId, int limit) {
    Optional<List<Post>> cached = cache.get(viewerId);
