design patternsoverview
what a pattern actually is

A named solution to a recurring problem.

A pattern is not code you copy. It is a shape you recognize: a proven arrangement of a few classes, plus a name so a whole team can point at it in one word.

The 1994 book Design Patterns, by the four authors everyone calls the Gang of Four, catalogued twenty-three. We take the handful that earn their keep in everyday backend code, then name the rest.

one warning, up front

Patterns are discovered, not forced.

You do not wake up and decide "today I will use a Factory." You feel a specific pain, and the pattern is the relief.

So reach for a pattern only when the problem it solves is the problem you actually have. Keep this in your pocket: every pattern today is just yesterday's SOLID, wearing a specific shape.

family 1 / creational

Who decides which class gets built, and where?

Scatter new across the code, and every change means hunting it down.

creationalquestion
factory

Ask for what you need. Don't build it yourself.

  • Instead of new scattered everywhere, callers ask a factory by name or type.
  • It hands back the object typed as an interface. The caller never names the concrete class.
  • Add a new product, change one factory. Every caller just works — dependency inversion, made concrete.
creationalfactory
factory · in code, and in the wild

You call one every time you open a database.


Beverage create(String name) {
    return switch (name.toLowerCase()) {
        case "espresso" -> new Espresso();
        case "tea"      -> new Tea();
        default -> throw new IllegalArgumentException(name);
    };
}


// JDBC: the classic factory
Connection c =
    DriverManager.getConnection(url);
// you get a Connection (interface).
// Postgres? MySQL? You never name the
// concrete driver class - the factory
// picks it from the url.

You already use it: DriverManager.getConnection, Calendar.getInstance(), List.of(...). Each hands back an interface and hides which concrete class it built.

creationalfactory · jdbc
builder

When a constructor grows ten arguments.

new Pizza(true, false, true, 12, false). Which true is the cheese? Nobody can read it, and it is easy to swap two by accident.

A builder lets you name each part and read like a sentence: .size(12).cheese().build(). Order-free, and it can validate before handing you a finished, valid object.

You already use it: StringBuilder — new StringBuilder().append("a").append(1).toString(). Also Stream.collect, and HttpRequest.newBuilder().

creationalbuilder
a two-minute aside you'll be asked in interviews

StringBuilder vs StringBuffer.

  • Same API. StringBuffer came first, and every method is synchronized — thread-safe, but it pays a lock on every call.
  • StringBuilder dropped the locks — not thread-safe, and faster. It is the right default.
  • Why is a default even safe? Because a builder used inside one method is a local variable.

Local = safe. Each thread has its own stack. A local variable lives there, so no other thread can even see it — nothing to race.

static / shared = not safe. A static field is one slot every thread sees. Two threads mutating it race — you need StringBuffer, a lock, or a concurrent type.

Rule of thumb: build strings in a local StringBuilder; only reach for synchronization when the state is genuinely shared across threads.

creationalthread safety
singleton, and a warning

Exactly one instance. Handle with care.

  • One instance, one global access point. Fine for a genuinely single thing: a config registry, a connection pool.
  • But it is global mutable state wearing a nice name. It hides dependencies and it makes tests hard.
  • Usually, yesterday's answer is better: create it once and inject it, rather than reach for it.

You already use it: Runtime.getRuntime(), a logger, your app's connection pool. One instance, reached from anywhere.

family 2 / structural

How do you fit objects together, especially when they don't match?

Structural patterns are about composition: wrapping, bridging, and simplifying.

structuralquestion
adapter

The travel plug for code.

  • You have a class the caller cannot use directly: a third-party client, a legacy API.
  • The adapter implements the interface the caller expects, and translates each call to the other side.
  • Neither side changes. The adapter absorbs the mismatch.
structuraladapter
adapter · in code, and in the wild

Scale a 0–100 dial onto a 0–11 amp.


class AmpAdapter implements SoundSystem {
    private final VintageAmp amp;     // 0..11 dial
    AmpAdapter(VintageAmp amp){ this.amp = amp; }

    public void setVolume(int pct) {  // 0..100 in
        amp.dial(pct * 11 / 100);     // ...scaled out
    }
    public void on() { amp.powerUp(); }
}

You already use it: InputStreamReader adapts a byte InputStream into a char Reader. Arrays.asList(arr) adapts an array into a List.

The caller speaks SoundSystem; the amp never learns a new trick. The adapter is the only thing that knows both sides.

structuraladapter · code
decorator

Add behaviour by wrapping, not editing.

A decorator implements the same interface as the thing it wraps, holds one, and adds its bit before or after delegating.

Because it is the same interface, you can wrap a wrap a wrap: a base call, wrapped in logging, wrapped in retries. New behaviour is a new wrapper, never an edit. Pure open-closed.

structuraldecorator
decorator · in code, and in the wild

A stream wrapping a stream.


class Milk implements Beverage {
    private final Beverage base;   // wraps one
    Milk(Beverage base){ this.base = base; }
    public String description(){
        return base.description() + ", milk";
    }
    public int cents(){ return base.cents() + 50; }
}


// java.io is decorators all the way down:
Reader r =
  new BufferedReader(     // adds buffering
    new InputStreamReader(  // adapts bytes
      new FileInputStream(f)));  // the base

You already use it: BufferedInputStream, Collections.unmodifiableList, synchronizedList — each wraps and adds one behaviour.

structuraldecorator · java.io
facade

One clean door over a messy room.

  • A subsystem of many classes, with a fiddly call order, and you want callers to have one simple entry point.
  • The facade is a single class whose only job is to orchestrate the subsystem behind one method.
  • Callers depend on the facade, not the tangle. Single responsibility at the boundary.

You already use it: SLF4J is literally a logging facade over Log4j/Logback; Spring's JdbcTemplate is one clean door over raw JDBC's open-query-map-close dance.

structuralfacade · slf4j
family 3 / behavioral

How do objects divide the work and talk at runtime?

Behavioral patterns are about the flow of decisions and messages between objects.

behavioralquestion
strategy

Swap the algorithm, not the code.

  • You met this yesterday: a family of interchangeable algorithms behind one interface.
  • The object holds the interface; you hand it a different implementation to change what it does.
  • Different sort order, pricing rule, discount: swap the object. Strategy is polymorphism on purpose.

You already use it: Comparator. list.sort(Comparator.comparing(User::age)) — you hand sorting a strategy, and it never changes.


class PercentOff implements Discount {
    public int applyTo(int c){ return c - c*pct/100; }
}
behavioralstrategy · comparator
observer

One changes, many react.

  • One object changes and several others must respond — without the first one knowing who they are.
  • The subject keeps a list of observers and notifies them all when something happens.
  • New listeners subscribe without the subject changing.

You already use it: every UI button's addActionListener; PropertyChangeListener; a Kafka topic fanning one event out to many consumers.

behavioralobserver · listeners
template method

Fixed skeleton, one step left open.

An algorithm with a fixed shape but one or two varying steps. The base class writes the skeleton as a final method and leaves the varying steps as hooks.

The order is locked; only the gap changes. The older, inheritance-based way to be open-closed — perfect when the shape is genuinely fixed.

You already use it: AbstractList writes the skeleton, you fill get() + size(); a servlet's service() calls your doGet/doPost.


abstract class Appliance {
    public final List<String> startUp() {
        var log = new ArrayList<String>();
        log.add("plugged in");   // fixed
        log.add(warmUp());       // the hook
        log.add("ready");        // fixed
        return log;
    }
    protected abstract String warmUp();
}
class Projector extends Appliance {
    protected String warmUp(){ return "warming bulb"; }
}

seven more, briefly

Nine you reach for. Seven more to recognise.

These turn up less often, and the third capstone (the text editor) is where you'd actually build them. We'll keep each to one clear idea — enough to recognise it in the wild.

Don't worry about mastering all seven today. Know the name and the one-line pain each one solves.

the other sevenrecognise, don't force
command

Wrap an action as an object.

  • Pain: you want to trigger, queue, log — and above all undo and redo — actions, without the caller knowing their details.
  • Shape: an interface with execute() and undo(). Keep an undo stack and a redo stack.

In the wild: every Undo button; a job queue's tasks; Runnable handed to a thread pool.
Editor · part 3: InsertText is a Command; the editor records and replays it.

behavioralcommand
memento

Snapshot state, restore it later.

  • Pain: undo needs the previous state back — but you don't want to leak the object's internals to get it.
  • Shape: the object hands out an opaque snapshot, and later restores itself from one. Only it knows what's inside.

In the wild: a game's save file; a database transaction's savepoint; version-control checkpoints.
Editor · part 3: snapshot() captures the buffer, restore() puts it back.

behavioralmemento
state

Behaviour that changes with mode.

  • Pain: what an action does depends on a mode, and the code is drowning in if (mode == ...) everywhere.
  • Shape: make each mode an object behind one interface. The context delegates to its current state.

In the wild: a TCP connection (open/closed/listening); a media player (playing/paused).
Editor · part 3: the same keystroke inserts in INSERT and is ignored in READONLY — the mode decides, not an if-ladder.

behavioralstate
composite

Treat a tree like a single thing.

  • Pain: a part-whole hierarchy — a document is sections of paragraphs — and you want to treat a leaf and a whole branch the same way.
  • Shape: one Node interface. Leaves and containers both implement it; an operation just recurses.

In the wild: the file system (a file and a folder are both "paths"); the HTML/DOM tree; Swing components.
Editor · part 3: Paragraph is a leaf, Section holds Nodes; a whole document is just a Node.

structuralcomposite
iterator

Walk it without knowing its shape.

  • Pain: visit everything in a structure, in order, without the caller learning it's really a tree.
  • Shape: expose a sequence — hasNext() / next(), or Java's Iterable. The caller loops; the traversal stays hidden.

You already use it: every for (x : collection) loop is the Iterator pattern.
Editor · part 3: walk every Paragraph depth-first; the tree looks like a flat list to the reader.

behavioraliterator
visitor · the hard one, taken gently

New operations, without touching the classes.

  • Pain: you keep needing new operations over a node tree — word count, spell-check, export — and editing every Node class each time is miserable.
  • Shape: nodes have accept(visitor), which calls visitor.visitParagraph(this). A new operation is a new visitor; the nodes never change.

The trade-off to remember: Visitor makes new operations easy and new node types hard. That's the whole point of it — and why it's the last one you reach for.

If this one doesn't fully land today, that's fine — it's the trickiest of the twenty-three. Recognise the name; build it only if you get to part 3.

behavioralvisitor
chain of responsibility

A pipeline of handlers.

  • Pain: a request should pass through independent steps, each doing its bit or handing it on — and you want to add, drop, or reorder steps freely.
  • Shape: each handler holds the next one; it does its work, then delegates down the chain. Order is just wiring.

You already use it: a Servlet filter chain; middleware in Express/Spring; try/catch handlers.
Editor · part 3: trim → collapse-spaces → censor, each a TextFilter passing text on.

behavioralchain of responsibility

the connection to yesterday

Patterns are SOLID, wearing shapes.

  • Strategy, Observer, Decorator, Factory: extend by adding. That is open-closed and dependency inversion.
  • Facade owns one workflow: single responsibility at a boundary.
  • Template Method locks a skeleton and trusts its hooks: Liskov, and open-closed by inheritance.
  • Adapter and the small role interfaces everywhere: interface segregation.

They are not a separate subject. They are the principles, named.

the failure mode

Patternitis: forcing patterns where there is no pain.

smells

A factory that builds one thing. A strategy interface with a single implementation. Five layers of indirection to read one config value.

the rule

Every pattern is indirection, and indirection costs a file, a hop, a name to learn. See the problem happen, then reach for the pattern. Not before.

Five moves, four patterns.

Part 1 · solution

In code: Factory & Decorator.

Factory — ask by name; the call site never writes new Espresso().

capstone · part 1solution 1/2
Part 1 · solution

In code: Strategy & Observer — three patterns, one line.

Strategy — one method behind an interface; swap the implementation.


final class PercentOff implements Discount {
    private final int percent;
    PercentOff(int p) { this.percent = p; }
    public int applyTo(int cents) {
        return cents - (cents * percent / 100);
    }
}

Observer — the Counter notifies every subscriber, and never knows who.


int placeOrder(Beverage drink, Discount discount) {
    int price = discount.applyTo(drink.cents()); // decorator+strategy
    for (OrderObserver o : observers)             // observer
        o.onOrder(drink.description(), price);
    return price;
}

placeOrder prices the decorated drink, applies the strategy, and fans out to observers — three patterns meeting in one method.

Part 3 · text editor · stretch

Seven patterns — for when you've nailed the first two.

  • Command + Memento — real undo and redo, and buffer snapshots. (CP1–2)
  • State — INSERT vs READONLY: the mode decides what a keystroke does. (CP3)
  • Composite + Iterator — a document tree of sections and paragraphs, walked as a flat list. (CP4–5)
  • Visitor — a word-count over the tree, without touching the node classes. (CP6, the hard one)
  • Chain of Responsibility — a trim → collapse → censor text pipeline. (CP7)

This is the tough module. If you get here, take it one checkpoint at a time — and don't let Visitor derail you.

capstone · part 3the shape
Part 3 · solution

In code: Command, Memento, State.


// COMMAND: do AND undo; two stacks in the editor
void run(Command c){ c.execute(); undo.push(c); redo.clear(); }
void undo(){ Command c = undo.pop(); c.undo(); redo.push(c); }
void redo(){ Command c = redo.pop(); c.execute(); undo.push(c); }

// MEMENTO: opaque snapshot of the buffer
Memento snapshot(){ return new Memento(buf.toString()); }


// STATE: the mode decides, no if-ladder
void type(char c){ mode.type(this, c); }

class InsertMode implements Mode {
    public void type(Editor ed, char c){ ed.buf.append(c); }
}
class ReadOnlyMode implements Mode {
    public void type(Editor ed, char c){ /* ignored */ }
}
capstone · part 3solution 1/2
Part 3 · solution

In code: Composite, Visitor, Chain.


// COMPOSITE: leaf and branch share one interface
interface Node { int wordCount(); void accept(NodeVisitor v); }
class Section implements Node {
    List<Node> children;
    public int wordCount(){
        int sum = 0;
        for (Node n : children) sum += n.wordCount(); // recurse
        return sum;
    }
}


// VISITOR: node calls back the matching method
public void accept(NodeVisitor v){ v.visitParagraph(this); }

// CHAIN: each filter does its bit, passes it on
final String process(String t){
    String out = apply(t);
    return next == null ? out : next.process(out);
}

Visitor's accept/visitX pair is "double dispatch" — the node picks the method. Read it slowly; it's the one that takes a second pass.
