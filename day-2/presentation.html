day 1 solved · the bank (1 of 3)

Account: the balance can never go illegal.


public final class Account {
    private long balanceCents;                 // private: only these methods touch it

    public Account(String id, long opening) {
        if (opening < 0) throw new IllegalArgumentException("negative opening");
        this.balanceCents = opening;
    }
    public void deposit(long c) {
        if (c <= 0) throw new IllegalArgumentException("must be positive");
        balanceCents += c;
    }
    public void withdraw(long c) {
        if (c <= 0)           throw new IllegalArgumentException("must be positive");
        if (c > balanceCents) throw new IllegalArgumentException("insufficient funds");
        balanceCents -= c;
    }
}

Encapsulation. The field is private, so the "never negative" rule lives in one place and no caller can route around it. Four tests: negative opening throws; deposit adds; non-positive throws; over-withdraw throws and leaves the balance untouched.

day 1 solved · the bank (2 of 3)

Transactions apply themselves.


sealed interface Transaction permits Deposit, Withdraw, Transfer {
    void apply(Ledger ledger);
}
record Deposit (String id, long cents) implements Transaction {
    public void apply(Ledger l) { l.account(id).deposit(cents); }
}
record Withdraw(String id, long cents) implements Transaction {
    public void apply(Ledger l) { l.account(id).withdraw(cents); }
}
record Transfer(String from, String to, long cents) implements Transaction {
    public void apply(Ledger l) {
        l.account(from).withdraw(cents);   // if this throws, no money has moved
        l.account(to).deposit(cents);
    }
}

Abstraction + polymorphism. A sealed family — the only three kinds — each implementing apply its own way. Transfer withdraws first, so a failure moves nothing.

day 1 solved · the bank (3 of 3)

The Ledger owns the accounts.


public final class Ledger {
    private final Map<String, Account> accounts = new HashMap<>();

    public void open(Account a) { accounts.put(a.id(), a); }
    public Account account(String id) {
        Account a = accounts.get(id);
        if (a == null) throw new NoSuchElementException("no account: " + id);
        return a;
    }
    public void run(Transaction t)           { t.apply(this); }          // polymorphic dispatch
    public void runAll(List<Transaction> ts) { for (var t : ts) run(t); }  // one loop, no type checks
}

Composition. The ledger holds a Map — it does not extend one — so no map methods leak. run just says t.apply(this) and lets the transaction decide. That one loop over a mixed list is the polymorphism test going green.

day 1 solved · the report (1 of 2)

Two formats, one contract.


final class CsvFormat implements ReportFormat {
    public String format(Report r) {
        if (r.rows().isEmpty()) return r.title();
        return r.title() + "\n" + String.join("\n", r.rows());
    }
}
final class JsonFormat implements ReportFormat {
    public String format(Report r) {
        StringJoiner rows = new StringJoiner(",", "[", "]");
        for (String row : r.rows()) rows.add("\"" + row + "\"");
        return "{\"title\":\"" + r.title() + "\",\"rows\":" + rows + "}";
    }
}

Liskov + interface segregation. Both implement the same one-method ReportFormat, so either is usable anywhere the interface is expected — and the interface is tiny, exactly one method.

The service injects a format, and validates.


class ReportService {
    private final ReportFormat format;              // depend on the interface (DIP)
    ReportService(ReportFormat format) { this.format = format; }

    String export(Report r) {
        if (r.title() == null || r.title().isBlank())   // one home for the rule (SRP)
            throw new IllegalArgumentException("title must not be blank");
        return format.format(r);                    // then just delegate
    }
}

ReportFormat plain = report -> "plain:" + report.title();   // a whole new format, one line
new ReportService(plain).export(r);                          // service UNCHANGED  (OCP)

DIP + SRP + OCP. It depends on the interface, validates in one place, and a brand-new format works with the service untouched.

The exporter injects a format and delegates.


final class StatementExporter {
    private final StatementFormat format;                 // injected from outside (DIP)
    StatementExporter(StatementFormat format) { this.format = format; }
    public String export(List<Account> accounts) {

final class CsvFormat implements StatementFormat {        // a new format = a new class
    public String format(List<Account> accounts) {
        StringJoiner out = new StringJoiner("\n"); out.add("id,balance");

what a pattern actually is

A named solution to a recurring problem.

Not code you copy. A shape you recognize: a proven arrangement of a few classes, plus a name so a whole team can point at it in one word.

The 1994 book Design Patterns, by the four authors everyone calls the Gang of Four, catalogued twenty-three. We take sixteen you actually build today, grouped into three families, and name the rest.

family 1 / creational · how objects are made

Who decides which class gets built, and where?

Scatter new SomethingConcrete() across the code, and every change to how it's built means hunting down every place that built it.

Creational patterns move that decision into one place — Factory, Builder, Singleton.

creationalquestion
factory

Ask for what you need. Don't build it yourself.

  • Callers ask a factory — by name, type, or config — instead of writing new.
  • It hands back the object typed as an interface. The caller never names the concrete class, so it never depends on it.
  • The choice of implementation lives in one place. Change it, add to it, and every caller just works — that is dependency inversion made concrete.

You build one in the capstone — so instead of the answer, look at three you call constantly.

creationalfactory
factory · what actually happens inside

One call, a different concrete class each time.

DriverManager
.getConnection(url)

Drivers register themselves on load. DriverManager walks its registered list, asks each "do you accept this URL?", and the first that says yes returns its Connection — a PgConnection for a postgres URL, a MySQL one for a mysql URL. You only ever see the Connection interface.

Calendar
.getInstance()

Reads your locale and returns a matching subclass: usually GregorianCalendar, but a Thai locale gets BuddhistCalendar, a Japanese-imperial locale gets JapaneseImperialCalendar. Same call — the concrete class depends on the environment, not on you.

List.of(...)

Picks an implementation by size: List.of() returns a shared empty list, one or two elements get a compact List12, more get a ListN — all hidden behind List. It optimises the object for the input, and you never know or care.

That's the whole point: same call site, different concrete object, decided by input or environment. The caller depends only on the interface — swap the guts and nothing upstream notices.

creationalfactory · internals
builder

When a constructor grows ten arguments.

new Pizza(true, false, true, 12, false). Which true is the cheese? Nobody can read it, and swapping two flags compiles fine and ships a bug.

A builder names each part and reads like a sentence — .size(12).cheese().build(). Order-free, skips the parts you don't set, and can validate before it hands you a finished, always-valid object.

You use it constantly: StringBuilder — append().append().toString(). Also Stream.collect, HttpRequest.newBuilder(), Stream.of(...).toList().

creationalbuilder
a two-minute aside interviewers love

StringBuilder vs StringBuffer.

  • Same API. StringBuffer came first — every method is synchronized, so it's thread-safe but pays a lock on every call.
  • StringBuilder dropped the locks — not thread-safe, and faster. It is the right default, and here's why that default is safe.

Local = safe. Each thread has its own stack. A local variable lives there — no other thread can even see it, so there is nothing to race over. A builder used inside one method is local.

static / shared = not safe. A static field is one slot every thread sees. Two threads writing it race. That is when you need a StringBuffer, a lock, or a concurrent type.

Rule of thumb: build in a local StringBuilder; synchronize only state that is genuinely shared across threads.

creationalthread safety
singleton, and a warning

Exactly one instance. Handle with care.

  • One instance, one global access point. Fine for a genuinely single thing: a config registry, a connection pool, the runtime.
  • But it is global mutable state wearing a nice name. It hides dependencies — nobody sees it in a constructor — and it makes tests hard, because you cannot swap it out.
  • Usually yesterday's answer is better: create it once at startup and inject it, rather than reach out and grab it.

In the JDK: Runtime.getRuntime() returns the one runtime; a logger factory hands back one shared logger per name. Private constructor, one static instance, one accessor.


private static final Hub INSTANCE = new Hub();
private Hub() { }                       // nobody else can make one
public static Hub get() { return INSTANCE; }

family 2 / structural · how objects compose

How do you fit objects together, especially when they don't match?

All about composition — the has-a from Day 1. Wrap something, bridge a mismatch, hide a mess, or treat a tree as one thing.

Adapter, Decorator, Facade, Composite.

structuralquestion
adapter

The travel plug for code.

  • You have a class the caller can't use directly — a third-party client, a legacy API with the wrong names or units.
  • The adapter implements the interface the caller expects, holds the awkward object, and translates each call across.
  • Neither side changes. The adapter absorbs the mismatch.

In the JDK: InputStreamReader adapts a byte InputStream into a char Reader. Arrays.asList(arr) adapts a raw array into a List. Both let old shapes flow into new APIs.


class AmpAdapter implements SoundSystem {   // expected API
    private final VintageAmp amp;           // 0..11 dial
    public void setVolume(int pct) {        // 0..100 in
        amp.dial(pct * 11 / 100);           // ...translated out
    }
}
structuraladapter
decorator

Add behaviour by wrapping, not editing.

A decorator implements the same interface as the thing it wraps, holds one, and adds its bit before or after delegating.

Same interface in, same interface out — so you can wrap a wrap a wrap: a base call, wrapped in buffering, wrapped in compression. New behaviour is a new wrapper, never an edit. Pure open-closed.

java.io is decorators all the way down:
new BufferedReader(new InputStreamReader(new FileInputStream(f))) — each layer adds one thing. Also Collections.unmodifiableList, synchronizedList.

structuraldecorator
facade

One clean door over a messy room.

  • A subsystem of many classes with a fiddly call order, and you want callers to have one simple entry point.
  • The facade is a single class whose only job is to orchestrate that subsystem behind one method.
  • Callers depend on the facade, not the tangle. Single responsibility at the boundary — the facade owns the workflow.

The name is literal in the wild: SLF4J is a logging facade over Log4j / Logback — you code to one API, swap the backend freely. Spring's JdbcTemplate is one door over JDBC's open-query-map-close dance.

structuralfacade
composite

Treat a tree like a single thing.

  • A part-whole hierarchy — a document is sections of paragraphs, a folder holds files and folders — and you want a leaf and a whole branch treated the same.
  • One Node interface. Leaves and containers both implement it; a container holds child Nodes, and an operation just recurses.
  • Any code that takes a Node takes the whole tree — no special-casing leaves versus branches.

You navigate these daily: the file system (a file and a folder are both "paths"); the HTML/DOM tree; Swing/Android view trees. size() on a folder just sums its children.


interface Node { int wordCount(); }
class Section implements Node {         // a container
    List<Node> children;
    public int wordCount() {
        int sum = 0;
        for (Node n : children) sum += n.wordCount();  // recurse
        return sum;
    }
}
structuralcomposite
family 3 / behavioral · how objects interact

How do objects divide the work and talk at runtime?

The biggest family — about the flow of decisions and messages between objects.

Strategy, Observer, Template Method — and the five behind the editor capstone: Command, Memento, State, Iterator, Visitor, Chain of Responsibility.

behavioralquestion
strategy

Swap the algorithm, not the code.

  • You met it yesterday: a family of interchangeable algorithms behind one interface.
  • The object holds the interface; you hand it a different implementation to change what it does. Sort order, pricing, discount — swap the object, not the code.
  • Strategy is polymorphism on purpose — the purest open-closed.

The everyday one: Comparator. list.sort(Comparator.comparing(User::age)) hands the sort a strategy — the sort algorithm never changes, only the comparison. Also a ThreadPoolExecutor's rejection policy.


users.sort(Comparator.comparing(User::name)   // one strategy
                     .thenComparing(User::age)); // composed
behavioralstrategy
observer

One changes, many react.

  • One object changes and several others must respond — without the first one knowing who they are.
  • The subject keeps a list of observers and notifies them all when something happens. New listeners subscribe without the subject changing.
  • It is how decoupled, event-driven code is wired.

You already use it: every UI addActionListener; PropertyChangeListener; a Kafka topic fanning one event to many consumers; the reactive Flow API.

behavioralobserver
template method

Fixed skeleton, one step left open.

An algorithm with a fixed shape but one or two varying steps. The base class writes the skeleton as a final method — so no subclass can reorder it — and leaves the varying steps as hooks.

The older, inheritance-based way to be open-closed. Perfect when the shape is genuinely fixed and only a step changes.

In the JDK: AbstractList writes the skeleton, you fill get() and size(). A servlet's service() calls your doGet/doPost.


abstract class Appliance {
    public final List<String> startUp() {  // locked order
        var log = new ArrayList<String>();
        log.add("plugged in");   // fixed
        log.add(warmUp());       // the hook
        log.add("ready");        // fixed
        return log;
    }
    protected abstract String warmUp();    // subclass fills this
}
behavioraltemplate method
command · behind every undo button

Wrap an action as an object.

  • Pain: you want to trigger, queue, log — and above all undo and redo — actions, without the caller knowing their details.
  • Shape: an interface with execute() and undo(). Each command captures what it needs both to do and to reverse its work. Keep an undo stack and a redo stack.

In the wild: every Undo/Redo; a job queue's tasks; a database transaction log; every Runnable you hand a thread pool is a command.


void run(Command c){ c.execute(); undo.push(c); redo.clear(); }
void undo(){ var c = undo.pop(); c.undo(); redo.push(c); }
void redo(){ var c = redo.pop(); c.execute(); undo.push(c); }
behavioralcommand
memento · command's partner

Snapshot state, restore it later.

  • Pain: undo needs the previous state back — but you don't want to leak the object's internals to get it.
  • Shape: the object hands out an opaque snapshot and later restores itself from one. Only it can read the snapshot's contents; everyone else just holds the token.

In the wild: a game's save file; a database transaction savepoint; a version-control checkpoint; an editor's undo history. Memento captures the "what", Command decides the "when".

Memento snapshot() captures the buffer; restore(m) puts it back — exactly one class can read inside m.

behavioralmemento
state

Behaviour that changes with mode.

  • Pain: what an action does depends on a mode, and the code is drowning in if (mode == ...) checks scattered everywhere.
  • Shape: make each mode an object behind one interface. The context delegates to its current state; a state can even hand back its successor.

In the wild: a TCP connection (listen / established / closed); a media player (playing / paused / stopped); an order (placed / paid / shipped). The state object decides — not an if-ladder.


void type(char c){ mode.type(this, c); }        // delegate to state
class InsertMode   implements Mode { public void type(Editor e,char c){ e.buf.append(c);} }
class ReadOnlyMode implements Mode { public void type(Editor e,char c){ /* ignore */ } }
behavioralstate
iterator · the one you use most

Walk it without knowing its shape.

  • Pain: visit everything in a structure, in order, without the caller learning it's really a tree, a ring buffer, or a paged list.
  • Shape: expose a sequence — hasNext() / next(), or implement Java's Iterable. The caller loops; the traversal stays hidden.

You use it every day: every for (x : collection) loop is the Iterator pattern — the compiler calls iterator(), then hasNext()/next() for you. Implement Iterable on your own tree and it walks like a flat list.

behavioraliterator
visitor · the tricky one, taken slowly

New operations, without touching the classes.

  • Pain: you keep needing new operations over a node tree — word count, spell-check, export to HTML — and editing every Node class each time is miserable.
  • Shape: nodes have accept(visitor), which calls back visitor.visitParagraph(this). The node picks the method — that two-step is "double dispatch." A new operation is a new visitor; the nodes never change.

// in each node:
public void accept(NodeVisitor v){ v.visitParagraph(this); }
// a new operation = a new visitor, nodes untouched:
class WordCount implements NodeVisitor {
    int total;
    public void visitParagraph(Paragraph p){ total += p.words(); }
}

The trade-off: Visitor makes new operations easy and new node types hard — the opposite of normal classes. That's why it's the last one you reach for. If it doesn't fully land today, recognise the name and move on.

behavioralvisitor
chain of responsibility

A pipeline of handlers.

  • Pain: a request should pass through a series of independent steps, each doing its bit or passing it on — and you want to add, drop, or reorder steps freely.
  • Shape: each handler holds the next one; it does its work, then delegates down the chain. The order is just wiring.

You've met it: a Servlet filter chain; middleware in Express or Spring; a logging framework's handler chain; even a stack of catch blocks. Add a step by inserting a link.


final String process(String t){
    String out = apply(t);                     // my step
    return next == null ? out : next.process(out); // hand on
}
// chain: trim -> collapse spaces -> censor

the connection to yesterday

Patterns are SOLID, wearing shapes.

  • Strategy, Observer, Decorator, Factory, Command: extend by adding. That is open-closed and dependency inversion.
  • Facade owns one workflow: single responsibility at a boundary.
  • Template Method locks a skeleton and trusts its hooks: Liskov.
  • Adapter and the small role interfaces everywhere: interface segregation.

Not a separate subject. The principles, named — which is why they clicked so fast.

design patternspatterns are solid
the failure mode

Patternitis: forcing patterns where there is no pain.

smells

A factory that builds one thing. A strategy interface with a single implementation. Five layers of indirection to read one config value.

the rule

Every pattern is indirection, and indirection costs a file, a hop, a name to learn. See the problem happen, then reach for the pattern. Not before.

The remaining seven of the twenty-three — Abstract Factory, Prototype, Bridge, Flyweight, Proxy, Mediator, Interpreter — are rarer in backend work. Recognise the names; look one up when you meet its problem.
